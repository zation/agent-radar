## 目标

在现有 `Release All` 两阶段发布链内，把 v0.4 P1 收集的生产 D1 当前投票和 GitHub `tool-feedback` Issue 转化为可回放、弱影响、受安全门禁约束的评分信号。

P2 必须保证：Issue 内容始终是不可信输入；LLM 只独立分类未处理的新 Issue；所有状态机、去重、计分、GitHub 回写和发布门禁均由确定性代码控制；反馈不能降低风险等级、提升 trust level 或绕过 critical safety gate。

## 范围

### 包含

- 读取生产 D1 当前投票的按 Tool 聚合快照。
- 读取 `zation/agent-radar` 中带 `tool-feedback` 标签的待处理 open Issue。
- 读取 closed + `feedback-accepted` 的历史评分 Issue。
- 确定性字段、Tool key、状态标签和重复校验。
- 对每个未处理新 Issue 发起一个隔离的 LLM 三态分类请求。
- accepted/rejected/needs-human-review/deprecated 状态机。
- `GitHub user ID + Tool key` 的 accepted Issue 最新记录去重。
- D1 `±0.2` 与 Issue `±1` 独立相加，Tool 调整限制为 `-3` 到 `+3`。
- `rating_result.v2` 的 base score、feedback adjustment 和 final score。
- reviewed bundle 内的 snapshot、classification、processing plan 和 summary artifacts。
- production approval 后、Worker deploy 前的幂等 GitHub comment/label/close 回写。
- Review Summary、manifest checksum、production evidence、测试和发布门禁接入。

### 不包含

- 新的 Web 管理后台。
- 在 D1 保存 Issue title、body、reason、comment 或其他自由文本。
- 让 LLM 调用 GitHub、D1 或任何工具。
- 多 Issue 批量 prompt。
- 根据反馈改变 risk level、trust level、来源证据或字段 provenance。
- 处理没有 `tool-feedback` 标签的普通仓库 Issue。
- 页面直接创建 Issue、复杂账号系统或企业权限治理。
- v0.5 Golden Query 与公开文档英文化。

## 核心原则

1. D1 当前投票与 accepted Issue 是两条独立信号流，不跨流去重。
2. 同一用户对同一 Tool 的 D1 当前投票和最新 accepted Issue可以合计贡献 `±1.2`。
3. Issue 流内部按 GitHub numeric user ID 与 Tool key 去重，只保留最新 accepted Issue。
4. 每个未处理 open Issue 最多自动分类一次；只有维护者移除 `feedback-needs-human-review` 后才允许重试。
5. 历史 accepted/rejected/deprecated Issue 不重新调用 LLM。
6. Build job 只生成不可变处理计划，不修改 GitHub；production approval 后才执行外部写操作。
7. 任一必要读取、分类、回写或反馈门禁失败都阻断发布。

## 输入选择

### D1 当前投票

构建期从生产 D1 `feedback_votes` 读取当前状态，并只输出每个 Tool 的：

- `up_count`
- `down_count`
- `row_count`
- snapshot timestamp
- canonical checksum

构建 artifact 不得包含 GitHub user ID、login 或逐用户 vote row。D1 schema 已保证同一 `(github_user_id, tool_id)` 最多一条当前投票，因此聚合可直接按 Tool 计分。

### 待处理 Issue

只选择：

- repository 为 `zation/agent-radar`
- state 为 `open`
- 带 `tool-feedback`
- 不带任何已完成处理标签
- 不带 `feedback-needs-human-review`

Issue 必须先通过确定性校验：

- Issue Form 字段完整且可唯一解析。
- Tool key 存在于当前 reviewed Tool Cards。
- vote 只能是 `up` 或 `down`。
- GitHub numeric user ID 存在。
- 状态标签不冲突。
- URL、title、body、comment 和用户名均只作为数据，不作为指令。

### 历史评分 Issue

只选择：

- state 为 `closed`
- 带 `feedback-accepted`
- 不带 `feedback-deprecated`

状态标签必须满足严格状态机：

- `accepted + rejected`：状态冲突，阻断发布。
- `accepted + needs-human-review`：状态冲突，阻断发布。
- `accepted + deprecated`：不参与评分；必须包含可验证的 replacement Issue 引用，否则阻断发布。
- 只有 rejected：不评分、不重新分类。
- 只有 deprecated：不评分、不重新分类。

本次新分类为 accepted 的 open Issue 与历史 accepted Issue共同进入 Issue 流去重和本次评分，不必等待下一次 release。

## 独立 LLM 分类

### 请求隔离

每个未处理 Issue 使用一个独立请求。实现固定最多 4 个并发，不得把多个 Issue 合并进同一个 prompt。单次 release 最多处理 50 个新 Issue；超过上限时阻断 build 并要求维护者分批处理，不能截断或静默跳过。

发送给 provider 的最小输入只包含：

- Issue number
- Tool key
- vote
- 用户填写的 reason
- 必要的 Tool identity 与公开安全边界摘要
- 固定分类规则版本

不得发送：

- OAuth token、Cookie、secret 或环境变量
- 其他 Issue 内容
- 完整 Tool Card、完整 reviewed bundle 或内部 prompt
- GitHub access token 或写权限

### 输出 schema

```json
{
  "decision": "accepted | rejected | needs-human-review",
  "reason_code": "valid_experience | invalid_context | insufficient_information | security_sensitive | conflicting_evidence",
  "summary": "short public-safe explanation"
}
```

约束：

- `feedback_classifier.v0.1` 固定 system prompt 与 schema，复用现有 provider registry、`AGENT_RADAR_LLM_API_KEY` 和 `AGENT_RADAR_LLM_MODEL`。
- `summary` 最多 240 个字符，只允许作为公开评论说明，不得包含原始 provider 响应。
- LLM 不获得工具调用能力。
- malformed output、provider timeout 或 transient provider error 最多重试一次。
- 单项最终失败阻断 build，不能静默降级或跳过。
- 历史 accepted/rejected/deprecated 不调用 LLM。
- `feedback-needs-human-review` 在标签未移除前不重复调用 LLM。

## Issue 状态机

`tool-feedback` 来源标签永久保留。正常处理状态互斥；唯一例外是旧 accepted Issue被废弃时保留 `feedback-accepted` 作为历史状态并追加 `feedback-deprecated`，此后 deprecated 优先且不再评分。处理状态为：

| 状态 | 标签 | 评论 | 最终状态 | 评分 |
| --- | --- | --- | --- | --- |
| accepted | `feedback-accepted` | 结论、reason code、release tag、规则版本 | closed | `±1` |
| rejected | `feedback-rejected` | 拒绝原因、release tag、规则版本 | closed | 0 |
| needs human | `feedback-needs-human-review` | 缺失信息或争议点、release tag | open | 0 |
| deprecated | `feedback-deprecated` | replacement Issue、release tag、规则版本 | closed | 0 |

维护者若希望重试 needs-human-review，必须先补充信息并移除 `feedback-needs-human-review`。下一次 Release All 才重新读取和独立分类。

### accepted Issue 去重

去重键：

```text
github_numeric_user_id + tool_id
```

排序使用 GitHub `created_at`，相同时间以更大 Issue number 为新。只保留最新 accepted Issue参与评分。所有旧 accepted Issue进入 deprecated processing plan，自动添加 `feedback-deprecated`、评论 replacement Issue 并关闭。后续构建直接跳过。

## 评分规则

规则版本：`feedback_rules.v0.1`。

```text
d1_adjustment    = (up_count - down_count) × 0.2
issue_adjustment = accepted_up_count - accepted_down_count
raw_adjustment   = d1_adjustment + issue_adjustment
applied_adjustment = clamp(raw_adjustment, -3, +3)
final_score      = clamp(base_score + applied_adjustment, 0, 100)
```

规则：

- 从第一条反馈开始计分，不设最小样本阈值。
- D1 与 Issue 独立相加；同一用户可同时贡献 D1 `±0.2` 和最新 accepted Issue `±1`。
- 用户取消 D1 投票后，D1 流不再贡献；其 accepted Issue仍按 Issue 流独立计算。
- D1 当前方向与 accepted Issue方向可以相反，按代数和计算。
- 中间计算使用精确十进制语义，artifact 和 API 最多保留一位小数。
- dimension scores 不变，避免把反馈伪装为某个评分维度。
- recommendation level 基于 final score 计算，但现有 evidence、maintenance、risk、trust 和安全上限继续优先。
- feedback 不能降低 risk、提升 trust、解除 avoid/consider 上限或绕过 critical safety gate。

## Rating Result v2

`rating_result.v2` 保留现有字段语义，并新增：

```json
{
  "schema_version": "rating_result.v2",
  "base_score": 84,
  "overall_score": 83.6,
  "feedback_adjustment": {
    "d1": 0.6,
    "accepted_issues": -1,
    "raw": -0.4,
    "applied": -0.4,
    "cap": 3,
    "rules_version": "feedback_rules.v0.1",
    "vote_snapshot_checksum": "sha256:...",
    "accepted_issue_ids": [123]
  }
}
```

Web、HTTP API、MCP、search index 和 recommendation 必须读取同一个 final `overall_score`。UI 有小数时显示一位，没有 adjustment 时可保持整数显示。

## 不可变 artifacts

### `feedback_vote_snapshot.v1`

保存 Tool 聚合计数、snapshot timestamp、总 row count、稳定排序和 checksum，不保存用户级数据。

### `feedback_classification.v1`

每个新 Issue 保存：

- Issue number 与公开 URL
- sanitized input checksum
- classifier rules version 与 model identifier
- decision、reason code、public-safe summary
- classification timestamp

不得保存完整 prompt、provider 原始响应或 secret。

### `feedback_processing_plan.v1`

保存 production approval 后要执行的：

- expected Issue `updated_at`
- expected original processing labels
- comment body 与隐藏幂等 marker
- labels to add/remove
- close/keep-open action
- deprecated replacement Issue number

### `feedback_summary.v1`

按 Tool 保存 D1 adjustment、Issue adjustment、raw/applied adjustment、accepted Issue IDs、规则版本和 snapshot checksum，并生成 Review Summary 聚合。

四类 artifacts 必须进入 reviewed bundle、artifact manifest checksum 和 production release evidence。

## 两阶段发布与 GitHub 回写

### Build reviewed bundle

Build job 权限只允许 `contents: read`、`issues: read` 和既有 artifact 权限。它完成读取、校验、独立分类、去重、评分、artifacts、rating/golden/safety/release gates，但不修改 GitHub。

### Production approval 后

Deploy job 获得 `issues: write`，先应用获批 bundle 中的 processing plan，再部署 Worker：

1. 重新读取目标 Issue。
2. 比较 Issue `updated_at` 和处理状态标签。
3. 若 approval 等待期间内容或状态发生变化，阻断部署，不覆盖新修改。
4. 查找隐藏幂等 marker；已应用同一 action 时视为成功。
5. 执行 comment、label 和 close/keep-open。
6. 任一必要回写失败则阻断 Worker deploy。
7. 回写全部成功后部署同一个 reviewed bundle。
8. 运行部署后 MCP smoke，生成 production release evidence。

隐藏 marker 示例：

```html
<!-- agent-radar-feedback:v0.1:issue-123:classification-sha256 -->
```

D1 在 build snapshot 后新增或切换的投票留到下一次 release，不改变已获批 bundle。build 后新建的 Issue同样留到下一次 release。

## 失败处理

以下情况阻断 build：

- GitHub Issue 读取失败。
- 生产 D1 snapshot 读取或 checksum 失败。
- 必需 LLM key/model 缺失。
- Issue 状态标签冲突。
- Issue Form 字段不可解析、Tool key 未知或 vote 非法。
- 单项分类在一次重试后仍失败。
- feedback artifact checksum、Rating Result v2 或 manifest 校验失败。
- feedback 调整导致 rating、golden query、critical safety 或 release gate 失败。

以下情况不阻断 build：

- 没有新 Issue。
- 没有 D1 投票或 accepted Issue；adjustment 为 0。
- 合法 `feedback-needs-human-review`；它不评分，保持 open，并进入 Review Summary。

以下情况阻断 deploy：

- processing plan 与当前 GitHub Issue `updated_at`/标签不一致。
- GitHub comment、label、close 或幂等检查失败。
- reviewed bundle、plan checksum 或 production evidence 不一致。

## 测试与验收

### 确定性单元测试

- Issue Form 解析、Tool key、vote 和状态标签校验。
- open/closed 输入选择与互斥状态冲突。
- accepted Issue 最新记录去重和 deprecated plan。
- D1 `±0.2`、Issue `±1`、独立相加、相反方向、`±3` cap 与 0–100 clamp。
- 一位小数精度和稳定排序/checksum。
- risk/trust/security gate 不被 feedback 提升。
- processing plan 幂等 marker。

### LLM contract 测试

- 每个新 Issue 一个独立请求，有限并发不改变隔离。
- 最小输入且不含 secret、Cookie、token、完整 Tool Card 或其他 Issue。
- 严格三态 schema、reason code 和 summary 限制。
- prompt injection fixture 不能改变 schema 或获得工具权限。
- malformed/provider/timeout 最多重试一次，最终失败阻断。
- 历史状态 Issue与未重置 needs-human-review 不调用 LLM。

### GitHub adapter 测试

- 固定 repository 与 `tool-feedback` 标签边界。
- comment、label、close、deprecated replacement 请求契约。
- approval 等待期间修改检测。
- workflow rerun 不重复评论或产生第二状态。
- 不创建 Issue、不修改 body、不访问其他仓库。

### Workflow 与完整验收

- Build job `issues: read`，Deploy job 仅在 production approval 后 `issues: write`。
- feedback artifacts 进入 reviewed bundle、manifest checksum 与 production evidence。
- GitHub 回写发生在 Worker deploy 前，失败阻断部署。
- Review Summary 展示样本量、最大调整、needs-human-review 和 deprecated 数量。
- 现有 24/24 golden queries、4/4 critical safety cases、API/MCP/Web 一致性继续通过。
- fixtures 覆盖 add/switch/cancel、accepted/rejected/human/deprecated 和 workflow rerun。
- 首个 P2 tag 完成真实 GitHub/D1 production approval、回写、部署与 smoke 演练。

## 完成标准

- 所有输入选择、分类、状态机、去重和计分规则有稳定 contract tests。
- 生产 D1 snapshot 不泄露用户级身份或投票行。
- 每个新 Issue独立分类一次，历史处理状态不重复调用 LLM。
- accepted/rejected/deprecated 自动评论并关闭，needs-human-review 评论后保持 open。
- Rating Result v2、search、recommendation、API、MCP 和 Web 使用同一 final score。
- feedback 不能削弱任何安全门禁。
- processing plan 只在 production approval 后幂等应用，外部状态变化或写失败阻断部署。
- 首个 P2 production release 生成完整 feedback evidence 并通过线上 smoke。
