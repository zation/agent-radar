# v0.5 P1：公开文档英文化设计

- 状态：已批准
- 实现提交：无
- 当前状态来源：[docs/15-roadmap.md](../../15-roadmap.md)

## 背景

Agent Radar 的公开 Web UI 已使用英文，但仓库入口、coding-agent 工作指令和当前领域权威文档仍以中文为主。公开用户、外部贡献者和 coding agent 需要在两种语言之间切换，也无法依靠自动检查阻止中文文案重新进入公开入口。

v0.5 采用 docs-first 顺序。P1 先完成公开文档英文化和严格语言门禁；P2 再迁移 Golden Query 源数据。两个阶段独立验收，避免大规模文档变更与推荐行为变化混在同一实现中。

## 目标

1. 将 `README.md`、`AGENTS.md` 和 `docs/00-product-brief.md` 至 `docs/14-web-ui.md` 全部改为英文。
2. 保持产品定位、当前实现事实、命令、路径、版本、发布证据、安全边界和文档职责不变。
3. 建立严格、可重复执行的公开语言检查，阻止汉字、CJK 标点和全角中文符号重新进入 P1 范围。
4. 保留中文 Roadmap 和冻结的历史 Spec/Plan，维持决策与执行历史的可追溯性。

## 文档迁移规则

- 翻译必须保留规范性强度；`must`、`must not`、`should` 和人工确认边界不能被弱化。
- 命令、环境变量、URL、文件路径、标识符、schema 名称、case ID、tag、commit SHA、run ID 和 deployment ID 原样保留。
- 当前实现事实以翻译前的 `README.md` 与 `docs/00-14` 为基线；不得借翻译机械引入未经验证的新事实。
- 发现明显过期事实时，只能在有仓库证据支持时同步修正；无法确认的内容保留原语义并登记后续事项。
- 文档间链接保持有效。标题翻译不得造成相对文件链接失效。
- P1 允许 Evaluation 页面继续展示中文 Golden Query 源数据；Roadmap 必须明确这是 P2 前的短暂过渡状态。

## 严格语言检查

新增一个无网络依赖的 TypeScript 检查模块和 CLI。检查器使用显式文件清单读取 P1 范围，而不是递归扫描整个 `docs/`，从结构上排除 Roadmap 与冻结历史文档。

检查规则覆盖：

- Han script 字符。
- CJK Symbols and Punctuation 区段中的中文标点。
- 全角中文符号。

检查不禁止英文技术文档中合理的其他 Unicode 字符，例如数学符号或英文排版符号。检查器不提供文本 allowlist；任何命中都必须改写为英文。失败输出包含 `file:line`、命中字符和最小上下文，并以非零状态退出。

提供 `npm run language:check`。该命令必须进入默认质量门禁，使本地测试和正式发布都不能绕过检查。检查模块与 CLI 分离，以便单元测试验证文件范围、字符识别和诊断格式。

## 数据流与组件边界

```text
explicit public document paths
  -> public-language validation module
  -> structured violations
  -> CLI diagnostics + exit status
  -> npm quality/release gates
```

- 检查模块只负责读取目标文本、定位违规字符并返回结构化结果。
- CLI 只负责格式化诊断和设置退出状态。
- `package.json` 只负责把检查命令接入现有门禁，不复制匹配规则。
- P1 不读取、改写或运行 Golden Queries，也不触碰推荐链路。

## 失败处理

- 检查命中：修正文案，不增加 allowlist，不缩小显式范围。
- 文档链接或命令验证失败：修复翻译造成的路径、锚点或代码块问题。
- 技术事实存在冲突：遵循 `AGENTS.md` 的文档优先级，领域事实以对应 `docs/00-14` 为准，阶段状态以 Roadmap 为准。
- 大规模翻译导致审阅困难：按职责拆分提交，但 P1 完成声明必须以全部目标文件通过严格检查为前提。

## 验证

P1 至少运行：

```bash
npm run language:check
npm test
npm run lint
npm run stylelint
npm run pages:build
```

另外人工抽查：

- README 的产品定位、快速开始、数据可信度、推荐安全和部署说明。
- AGENTS 的必读文档、权限边界、验证要求和最终输出格式。
- API/MCP 示例、shell 命令、环境变量、相对链接和发布证据。
- `docs/00-14` 之间的术语一致性与规范性措辞。

## 验收标准

- P1 显式范围内没有汉字、CJK 标点或全角中文符号。
- `npm run language:check` 对每个违规返回稳定的文件和行号，并以非零状态退出。
- Roadmap 与冻结 Spec/Plan 没有被机械翻译。
- README、AGENTS 和 `docs/00-14` 的技术事实、安全边界、命令和链接保持准确。
- 现有测试、lint、stylelint 和 Web 构建通过。
- Roadmap 将 P1 标记为完成、P2 标记为待开始；P1 不宣称整个 v0.5 完成。

## 风险与缓解

- 风险：翻译弱化安全或发布门禁。缓解：保留规范性强度并专项审查 `must`/`must not`/人工确认措辞。
- 风险：严格规则误伤合理 Unicode。缓解：只匹配 Han、CJK 标点和全角中文符号，不使用通用 ASCII-only 规则。
- 风险：大量文档 diff 隐藏事实变化。缓解：翻译不承担无关重写，只在有仓库证据时修正事实。
- 风险：P1 完成后 Evaluation 仍短暂显示中文。缓解：在 Roadmap 明示过渡状态，并以 P2 作为 v0.5 完成前置条件。
