# v0.8 P1: Compact Recommendation Prompt Design

- Status: Completed
- Implementation commits: `8d8c06c6`
- Current status source: [docs/15-roadmap.md](../../15-roadmap.md)

## Background

The `all-v0.7.1` production baseline sends the complete 76-card catalog to each of 24 Golden Queries and records 902,867 input tokens and 965,948 total tokens. Prompt construction preserves the intended normalized Tool Card and Rating fields but serializes the request JSON with two-space indentation. Local measurement of the same 76-card prompt found 163,104 bytes in the current representation and 119,952 bytes in compact JSON.

Two bounded MiniMax M3 cache probes were run before this design. The first kept one user message, moved the stable catalog before the changing query, and ran two requests sequentially; each request used about 26.6k input tokens but reported only 128 cached tokens, about a 0.48% hit rate. The second placed the same 119,756-byte catalog context in an identical system message and put only 171-197 bytes of changing query data in the user message. It reported 114 and 128 cached tokens, about 0.43% and 0.48%. All four requests returned HTTP 200 and valid JSON. The probes did not change repository code and show that the current MiniMax M3 route cannot be treated as a reliable catalog-reuse mechanism.

## Goals

1. Serialize recommendation prompts as compact JSON without changing their parsed structure, field order, catalog membership, provider request shape, or public recommendation contracts.
2. Add a regression test that proves the emitted prompt is the canonical compact serialization of its parsed value and still contains the original query and ordered Tool Card IDs.
3. Preserve the 24-query provider evaluation, four critical safety gates, retry policy, and `eval_token_usage.v1` evidence.
4. Use the next real-provider reviewed build to measure the actual token delta from `all-v0.7.1`.

## Non-Goals

- Top-K or semantic retrieval, catalog field removal, prompt batching, provider sessions, model changes, or output limits.
- Depending on MiniMax automatic or explicit prompt caching.
- Reducing the Golden Query suite or relaxing safety expectations.
- Adding a token-cost release threshold before a real-provider compact-prompt baseline exists.

## Design

`buildRecommendationPrompt` keeps the same JavaScript value and property ordering but calls `JSON.stringify(value)` without indentation. Provider adapters, recommendation normalization, deterministic safety, Eval Runner behavior, and public HTTP/MCP/Web schemas remain unchanged.

This is intentionally the first v0.8 increment because whitespace removal cannot exclude a candidate or alter a safety field. A later retrieval increment may consider a bounded shortlist only after it defines recall, diversity, safety-anchor, fallback, and provider A/B gates in a separate Spec.

## Acceptance

1. The compact-prompt regression test fails against indented serialization and passes after the implementation.
2. Parsed prompt query and ordered catalog Tool IDs remain unchanged.
3. Existing recommendation, API, MCP, Eval Runner, and token-usage tests pass.
4. `npm test`, `npm run lint`, `npm run stylelint`, `npm run pages:build`, and `git diff --check` pass.
5. Completion requires a real-provider 24/24 Golden Query run with 4/4 critical safety and valid token-usage evidence compared with the `all-v0.7.1` baseline.

## Risks

- JSON whitespace is normally semantically irrelevant, but model behavior can still vary because the token sequence changes. The real-provider gate remains mandatory.
- Byte reduction does not equal token reduction for every tokenizer. Only provider-reported usage establishes the production result.
- MiniMax caching may change later, but v0.8 P1 remains correct without it.
