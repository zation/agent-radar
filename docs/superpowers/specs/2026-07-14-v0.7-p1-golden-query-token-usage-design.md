# v0.7 P1: Golden Query Token Usage Observability Design

- Status: Completed
- Implementation commits: `5e9cfdc9`, `08bed60a`, `b053b226`, `09cb9168`, `f4f53290`
- Current status source: [docs/15-roadmap.md](../../15-roadmap.md)

## Background

Agent Radar runs 24 provider-backed Golden Queries during `Release All`. Each query currently sends the recommendation instruction, query, and compact catalog context to the selected provider, but the reviewed bundle records only evaluation outcomes. It does not preserve provider-reported input, cached-input, output, or total token usage, so maintainers cannot measure how Tool Card growth affects release cost and latency.

v0.7 begins with observability before catalog expansion. P1 records actual provider-reported usage without changing recommendation behavior or public contracts. P2 will use this evidence to choose and verify a bounded Tool Card expansion batch.

## Goals

1. Record normalized token usage for every provider request attempt made by the 24 Golden Queries.
2. Count retry consumption instead of recording only the final successful attempt.
3. Generate an immutable `eval_token_usage.v1` artifact bound to the release and reviewed bundle.
4. Surface totals, availability, retries, averages, and the highest-usage cases in the GitHub Actions review summary.
5. Keep missing or malformed provider usage non-blocking while making evidence corruption blocking.
6. Preserve the public Recommendation Result, HTTP, MCP, and Web contracts.

## Non-Goals

- Prompt caching, multi-query batching, or provider session APIs.
- Token budgets or cost-based release gates.
- Character-based or model-estimated token counts.
- Top-K retrieval before recommendation.
- An independent Data Build or scheduled ingestion workflow.
- Automatic production deployment.
- Tool Card expansion; that is v0.7 P2.

## Current Flow

`runGoldenQueries` executes cases with concurrency two. Each attempt calls `recommendTools`, which builds a compact catalog prompt and invokes the configured OpenAI-compatible provider adapter. The adapter parses recommendation JSON and discards any response-level `usage` object. The Eval Summary records pass/fail behavior, but neither it nor the preview manifest records provider consumption.

The existing retry policy may issue up to three requests for a case. A usage design that observes only the final Recommendation Result would undercount failed schema or transient-provider attempts.

## Design Overview

P1 adds an internal usage boundary beside recommendation output:

```text
Provider response
  -> recommendation output
  -> normalized provider usage

Golden Query attempt
  -> recommendation and deterministic safety
  -> Eval Result
  -> Eval Token Usage Collector

Eval Token Usage Collector
  -> eval_token_usage.v1
  -> reviewed-bundle manifest checksum
  -> GitHub Actions review summary
```

Usage is evaluation and release evidence. It is not part of the user-facing recommendation result.

## Internal Components

### Normalized Provider Usage

The provider adapter recognizes non-negative integer token fields from supported OpenAI-compatible response shapes:

- input: `prompt_tokens` or `input_tokens`;
- cached input: `prompt_tokens_details.cached_tokens` or `input_tokens_details.cached_tokens`;
- output: `completion_tokens` or `output_tokens`;
- total: `total_tokens`.

The normalized representation is:

```ts
interface NormalizedProviderUsage {
  status: "reported" | "unavailable";
  input_tokens: number | null;
  cached_input_tokens: number | null;
  output_tokens: number | null;
  total_tokens: number | null;
  unavailable_reason?: "missing_provider_usage" | "invalid_provider_usage" | "request_failed";
}
```

`reported` requires valid input, output, and total values. Cached input remains `null` when the provider omits it. P1 does not recompute or require `total_tokens === input_tokens + output_tokens`, because provider accounting semantics may differ.

Provider usage is untrusted external data. Negative numbers, fractional numbers, strings, non-finite values, and malformed objects normalize to `unavailable`; they do not change a valid recommendation into an evaluation failure.

### Attempt Context

The Eval Runner owns the stable attempt identity because it owns case concurrency and retries. Every actual provider call records:

- `case_id`;
- one-based `attempt`;
- provider and API model identifier;
- `outcome`: `completed`, `provider_error`, or `schema_error`;
- stable failure category when applicable;
- normalized usage.

A retry creates a new attempt record. Previously consumed tokens remain in the artifact. A no-key blocked evaluation makes no provider request and therefore records no attempt; the artifact represents the blocked suite explicitly rather than inventing zero-token provider responses.

### Eval Token Usage Collector

The collector is internal to evaluation and does not mutate `RecommendationResult`. It accepts attempt records from concurrent cases, then sorts them by `case_id` and attempt before building evidence. Totals are recomputed from `reported` attempts only.

`runGoldenQueries` keeps its current Eval Summary behavior. Pipeline orchestration supplies the collector and writes the separate usage artifact after evaluation, avoiding a public recommendation schema change.

Each case contains its ordered attempts, reported and unavailable counts, retry count, known token totals, and evaluation execution state. Summary totals must be reproducible from case attempts. The average is `null` when there are no reported attempts.

The reviewed-bundle manifest records the artifact path, checksum, schema version, and compact summary. Finalization validates the schema, order, release identity, arithmetic, and checksum. A missing or internally inconsistent artifact after an executed provider evaluation blocks finalization because release evidence is damaged.

When evaluation is blocked before any provider request, the artifact remains valid with zero attempts and an explicit blocked execution state. This preserves the existing evaluation failure while avoiding false usage data.

## GitHub Actions Summary

The existing reviewed-bundle summary gains a `Golden Query Token Usage` section with:

- provider and API model identifiers;
- case count and provider request attempts;
- reported and unavailable attempt counts;
- input, cached-input, output, and total tokens;
- retry count;
- average total tokens per reported attempt;
- the five highest-usage Golden Query cases;
- a warning when any attempt has unavailable usage.

The summary derives from the validated artifact in the reviewed bundle. It does not parse console logs and never displays prompts, response content, credentials, authorization headers, or raw provider error bodies.

## Failure and Trust Policy

- Missing provider usage is `unavailable` and does not block release.
- Malformed provider usage is `unavailable` and does not block release.
- Provider, schema, and quality failures retain their existing evaluation semantics.
- Retry attempts count toward request and token totals when usage is reported.
- Usage artifact schema, ordering, identity, checksum, or arithmetic failures block reviewed-bundle finalization.
- P1 observes consumption only; it introduces no token threshold or cost policy.

## Verification

Unit and integration coverage must prove:

1. Valid input, cached-input, output, and total fields normalize correctly.
2. Supported OpenAI-compatible aliases normalize to one representation.
3. Missing, negative, fractional, string, and malformed values become `unavailable`.
4. Provider and schema errors create attempt records without changing existing failure categories.
5. Retries preserve every attempt and include all reported consumption.
6. Two-case concurrency still produces deterministic `case_id + attempt` ordering.
7. Blocked-no-key evaluation creates zero provider attempts and explicit blocked state.
8. Artifact totals recompute from cases and reject tampering.
9. Pipeline and manifest include the artifact and checksum.
10. GitHub Summary renders totals, unavailable warnings, retries, and the five highest-usage cases.
11. Recommendation Result, HTTP, MCP, and Web contracts contain no usage fields.

Standard verification remains `npm test`, `npm run lint`, `npm run stylelint`, `npm run pages:build`, and `git diff --check`. A real-provider Release All run is required to verify production usage availability, but unit tests do not call a live provider or fabricate production evidence.

## Rollout and Follow-Up

P1 ships as observability only. After at least one real-provider release records usage, v0.7 P2 will define a scenario-driven Tool Card expansion batch and use the measured per-case and total consumption as planning evidence. Prompt optimization, independent Data Build, and scheduled bounded expansion remain separate future decisions.
