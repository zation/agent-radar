# v0.5 P2：Golden Query 英文化设计

- 状态：已完成
- 实现提交：`50ee6191`、`b64d3136`、`9aa196b3`、`928c0bbf`、`cde71124`、`defe3958`、`f4d97adc`、`2235d776`、`31c12891`、`c5b3a49b`、`ab498259`、`429c86f2`、`205819e2`
- 当前状态来源：[docs/15-roadmap.md](../../15-roadmap.md)

## 背景

Evaluation UI 直接消费 reviewed bundle 中的 `golden_queries.json`，并原样展示 `query.task` 和 `review_notes`。当前 UI chrome 为英文而这两个源字段为中文。P2 在 P1 完成公开文档英文化后迁移评测源数据，不在 UI 层增加临时翻译、双语副本或运行时 i18n。

Golden Query 文本同时是 LLM-backed 推荐输入。翻译可能改变模型理解和候选结果，因此 P2 必须把语义不变量、确定性安全断言和真实 provider eval 作为发布门禁。

## 目标

1. 将全部 24 条 Golden Queries 的 `query.task` 与 `review_notes` 改为英文。
2. 保持 case ID、case 数量、category、severity、expected、权限要求、任务约束和 critical safety gate 语义不变。
3. 更新对应 eval fixtures、测试、生成报告和权威文档示例，不维护 UI 翻译副本。
4. 扩展严格语言检查，使两个目标字段不能重新混入中文。
5. 通过真实 provider golden eval 证明迁移后仍为 24/24，critical safety 为 4/4。

## 范围

### 修改范围

- `src/eval/golden-queries.ts` 中 24 条 case 的 `query.task` 与 `review_notes`。
- Golden Query 不变量 fixture 与相关 contract tests。
- P1 建立的公开语言检查模块和 CLI。
- 直接引用 Golden Query 文本的 `docs/00-14` 示例、测试 fixture、快照或断言。
- 由 pipeline 重新生成但不作为长期源码提交的 eval artifacts 与报告。
- P2 完成时的领域权威文档和 Roadmap 状态。

### 明确排除

- EvalCase schema 变更。
- case ID、数量、category、severity 或 owner 的调整；`updated_at` 仅允许统一更新为 P2 实际迁移日期。
- `expected`、`risk_tolerance`、permissions、environment、tool type、tag 或 stack 约束变更。
- 推荐引擎、评分规则、安全动作上限和 release gate 放宽。
- UI 翻译表、运行时 i18n 或中英文双份 Golden Query。
- 冻结 Spec/Plan 的机械翻译。

## 翻译语义规则

- `query.task` 使用自然、明确、适合真实开发者输入的英文，不逐字制造生硬提示词。
- `review_notes` 必须保留原有推荐偏好、风险理由、权限提醒和人工确认要求。
- 对 `production`、`unknown source`、`directly modify`、`delete`、`live payment` 等安全关键限定词使用明确英文，不使用可能弱化风险的委婉表达。
- 相同安全场景采用一致术语，避免同一权限或动作在不同 case 中发生语义漂移。
- 只翻译两个目标字段；任何非文本约束变化都视为回归。
- 24 条 case 的 `updated_at` 统一更新为 P2 实际迁移日期，用于准确表达源数据更新时间；该时间字段不参与推荐或安全语义不变量。

## 不变量保护

新增一份不含 `query.task` 与 `review_notes` 的可审阅 fixture，记录每个 case 的保护投影：

- `id`
- `schema_version`
- `category`
- `query` 中除 `task` 外的全部字段
- `expected`
- `severity`
- `owner`

测试从运行时 `goldenQueries` 生成相同投影并执行深度比较。该 fixture 在文本翻译时不得更新；只有未来经过独立设计批准的评测语义变更才能修改。

现有 suite contract 继续要求：

- 总数恰好为 24。
- ID 唯一且稳定。
- critical case 恰好为 4。
- 任一 critical failure、缺失、未执行或 provider/schema error 阻断发布。

## 严格语言检查扩展

P1 的文档检查保持不变。P2 复用同一字符识别规则，并结构化加载 `goldenQueries`，只检查每条 case 的 `query.task` 和 `review_notes`。

失败诊断使用稳定字段路径，例如：

```text
src/eval/golden-queries.ts:gq-python-test-coverage.query.task contains prohibited CJK text
```

检查器不扫描或限制 Tool Card 的外部原始证据，也不把整个 TypeScript 源文件纳入中文禁令，以免误伤非公开内部注释；目标是保证公开 Evaluation 数据字段为英文。

## 数据流

```text
src/eval/golden-queries.ts
  -> invariant projection test
  -> strict target-field language check
  -> runGoldenQueries with existing recommendation + safety path
  -> golden_queries.json + eval_summary.json + report
  -> Evaluation UI direct rendering
```

UI 继续通过 `createEvaluationView` 把 `evalCase.query.task` 映射为 `task`、把 `evalCase.review_notes` 映射为 `why`。P2 不新增翻译函数，也不在浏览器改写 artifact 文本。

## 失败处理

- 不变量测试失败：恢复非文本字段，不更新 fixture 来掩盖差异。
- 语言检查失败：修正目标字段英文，不添加 allowlist。
- Contract test 或 UI test 失败：修复迁移造成的 fixture/断言偏移，不改变推荐安全规则。
- Provider eval 出现 `quality_failure`：先检查英文任务与 review note 是否改变原意，再审查 Tool Card、prompt 输入和实际候选；不得放宽 expected 或 critical gate。
- Provider eval 出现鉴权、限流、模型或 schema 错误：按环境或 provider 故障处理，不宣称推荐质量通过。
- 无可用真实 provider：P2 保持未完成状态，不能用 fake client 或 blocked eval 替代 24/24 验收。

## 风险与缓解

- 风险：英文措辞改变模型意图识别。缓解：使用自然任务语言、保护风险限定词，并强制真实 provider eval。
- 风险：翻译时顺手修改 expected。缓解：用独立保护投影 fixture 深度比较非文本语义。
- 风险：测试只验证文本没有汉字，却未验证 UI 数据源。缓解：保留并增强 Evaluation view contract，断言 UI 直接消费源字段。
- 风险：provider 非确定性造成偶发失败。缓解：保留失败分类和原始可追溯结果；只在完整 24/24 通过时完成 P2。

## 完成记录

24 条 Golden Query 的 `query.task` 与 `review_notes` 已全部迁移为英文，保护投影、稳定 ID、expected 和 4 个 critical gate 未变。实现期间发现 provider 空候选会覆盖未知来源代码执行的 `avoid` 上限，因此补充确定性安全优先级和 contract test；同时为真实 provider eval 增加有界并发、请求超时、退避重试和 release metadata 绑定。最终 MiniMax M3 provider eval、pipeline artifact 与 release check 均为 24/24，critical safety 4/4，artifact 绑定 `v0.5-p2-local` 和提交 `205819e2`。本 Spec 自此冻结。
