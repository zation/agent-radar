# v0.4 P1：GitHub 身份与 Tool 反馈采集

> 状态：已批准
>
> 实现提交：无
>
> 当前状态来源：[`docs/15-roadmap.md`](../../15-roadmap.md)、[`docs/14-web-ui.md`](../../14-web-ui.md)、[`docs/01-requirements.md`](../../01-requirements.md)

## 目标

在现有单 Cloudflare Worker + Static Assets 架构内完成 v0.4 P1 的真实反馈纵向链路：用户从 Tool Detail 点击 👍/👎，必要时通过最小 GitHub OAuth 登录，投票写入 D1，并可主动跳转 GitHub Issue Form 补充详细原因。

P1 只采集当前投票与结构化 Issue 入口，不实时修改评分；Issue 分类、反馈权重和 reviewed bundle 接入属于 v0.4 P2。

## 范围

### 包含

- GitHub OAuth Web Application Flow，只读取公开 `id` 与 `login`。
- 30 天 HMAC 签名 HttpOnly session cookie，不建立服务端 session 表。
- 未登录用户点击反馈后登录，callback 返回 Tool Detail 并自动完成原投票。
- D1 单用户、单 Tool Card 的唯一当前 `up`/`down` 投票。
- 匿名聚合票数、本人当前投票、切换与取消。
- Tool Detail 的 thumb up/down、聚合数量、失败状态和详细反馈 Dialog。
- GitHub Issue Form 与安全的预填 URL。
- 顶部 GitHub 身份入口与 Sign out。
- D1 binding、migration、Worker 配置、测试与部署文档。

### 不包含

- GitHub email、repository、organization 或其他 OAuth scope。
- OAuth token 持久化。
- D1 users/session 表或公开投票用户列表。
- Agent Radar 内的自由文本反馈。
- 页面或 Worker 直接创建 GitHub Issue。
- 一次投票实时调整 Rating。
- P2 Issue 三态分类、自动标签/评论/关闭或 `feedback_rules.v0.1`。

## 架构

现有 Worker 继续承载 Static Assets、只读 HTTP API 和 MCP。新增 auth/feedback 路由在 artifact repository 装配前分流；现有推荐、搜索、评分解释和 MCP 路由保持不变。

```text
Browser Tool Detail
  -> GET /api/feedback/tools/:tool_id
  -> PUT /api/feedback/tools/:tool_id/vote
       -> signed session cookie
       -> Origin + schema + Tool Card validation
       -> D1 feedback_votes

Anonymous thumb click
  -> GET /api/auth/github/start?tool_id=...&vote=...
  -> GitHub authorize (empty scope)
  -> GET /api/auth/github/callback
       -> validate OAuth state cookie
       -> exchange code
       -> GET GitHub /user
       -> discard access token
       -> D1 upsert original vote
       -> issue signed session cookie
       -> redirect to /?tool=<id>&feedback=<vote>
```

## Worker bindings 与配置

Worker `Env` 新增：

- `DB: D1Database`
- `GITHUB_OAUTH_CLIENT_ID: string`
- `GITHUB_OAUTH_CLIENT_SECRET: string`
- `AGENT_RADAR_SESSION_SECRET: string`

生产配置：

- `GITHUB_OAUTH_CLIENT_ID` 为普通 Text variable。
- `GITHUB_OAUTH_CLIENT_SECRET` 与 `AGENT_RADAR_SESSION_SECRET` 为 Worker Secret。
- `wrangler.toml` 声明 D1 binding 和 required secrets；Client ID 的非敏感值不写死到仓库。
- 本地开发使用 `.dev.vars`，文件保持 gitignored；测试注入假 GitHub transport 和内存 D1 adapter，不依赖真实 secret。

## 身份与 Cookie

### Session cookie

Cookie 名称：`agent_radar_session`。

HMAC-SHA256 签名 payload：

```json
{
  "v": 1,
  "github_user_id": "123456",
  "github_login": "octocat",
  "issued_at": 1783785600,
  "expires_at": 1786377600
}
```

- 有效期 30 天。
- 属性：`HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=2592000`。
- payload 只签名、不加密；不得包含 token、email 或私有资料。
- 验证签名使用常量时间比较，并拒绝未知版本、格式错误和过期 payload。
- `POST /api/auth/logout` 仅通过 `Max-Age=0` 清除 cookie，不操作数据库。

### OAuth state cookie

Cookie 名称：`agent_radar_oauth_state`。

- 有效期 10 分钟。
- 包含随机 nonce、Tool ID、`up`/`down` 意图、站内返回位置和过期时间，并使用同一 session secret 签名。
- callback 同时校验 query `state` 与 cookie nonce，成功或失败后均清除 state cookie。
- 返回位置只允许 Tools 页面及已验证 Tool ID，不接受外部 origin 或任意 URL。

## HTTP API

### `GET /api/auth/github/start`

Query：

- `tool_id`：当前 reviewed bundle 中存在的 Tool Card ID。
- `vote`：`up` 或 `down`。

行为：验证输入，签发短期 state cookie，302 到 `https://github.com/login/oauth/authorize`。不传 scope，使用配置的 callback URL。

### `GET /api/auth/github/callback`

行为：

1. 校验 `code`、query state 与 state cookie。
2. POST GitHub access token endpoint。
3. 使用 token GET GitHub `/user`，只读取 `id` 与 `login`。
4. 验证原始 Tool ID 仍存在。
5. 在一个明确的 D1 写入步骤中 upsert 原投票。
6. 丢弃 token，签发 session cookie，清除 state cookie。
7. 303 返回 `/?tool=<tool_id>&feedback=<vote>`。

任一步失败均不得写半成品 session；GitHub code/token/cookie 不进入日志。

### `GET /api/auth/session`

响应：

```json
{ "user": { "github_user_id": "123456", "github_login": "octocat" } }
```

未登录或 cookie 无效时稳定返回 `{ "user": null }`，并可清除无效 cookie。

### `POST /api/auth/logout`

清除 session cookie，返回 204。该接口不要求 D1。

### `GET /api/feedback/tools/:tool_id`

匿名响应：

```json
{ "tool_id": "mcp-playwright", "up": 12, "down": 2, "viewer_vote": null }
```

若请求带有效 session，`viewer_vote` 为 `up`、`down` 或 `null`。未知 Tool ID 返回 404；D1 不可用返回稳定 503，不伪造零票。

### `PUT /api/feedback/tools/:tool_id/vote`

Body：

```json
{ "vote": "up" }
```

`vote` 可为 `up`、`down` 或 `null`。要求：

- 有效 session，否则 401。
- `Origin` 与当前 Worker origin 完全一致，否则 403。
- `Content-Type: application/json`，body schema 严格校验。
- Tool ID 存在，否则 404。
- 同一用户固定一分钟窗口最多 30 次 mutation，超限返回 429。

`up`/`down` 使用 upsert；`null` 删除当前记录。重复写相同值保持幂等。响应返回新的聚合数与 viewer vote。

## D1 数据模型

新增 migration `0002_feedback_votes.sql`：

```sql
CREATE TABLE feedback_votes (
  github_user_id TEXT NOT NULL,
  github_login TEXT NOT NULL,
  tool_id TEXT NOT NULL,
  vote TEXT NOT NULL CHECK (vote IN ('up', 'down')),
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  PRIMARY KEY (github_user_id, tool_id)
);

CREATE INDEX idx_feedback_votes_tool_vote
  ON feedback_votes (tool_id, vote);

CREATE TABLE feedback_rate_limits (
  github_user_id TEXT PRIMARY KEY,
  window_started_at TEXT NOT NULL,
  mutation_count INTEGER NOT NULL CHECK (mutation_count >= 0)
);
```

- GitHub user ID 作为稳定身份，login 仅作为当前公开显示元数据。
- P1 不保存历史投票、自由文本、OAuth token 或 email。
- `feedback_rate_limits` 只实现 vote mutation 的每用户固定一分钟窗口；过期窗口在下一次请求时原位重置，不保存通用访问日志。

## 前端状态与交互

### 身份

- App 启动加载 `/api/auth/session`。
- 顶栏右侧 release tag 附近显示：未登录 `Sign in`；已登录 GitHub username。
- 身份菜单只提供 `Sign out`。
- 身份加载失败不阻断只读 Tools/Evaluation。

### Tool Detail 反馈

- 反馈区位于总分/风险下方，高于 Decision signals 与 Evidence。
- 只显示两个语义明确的 thumb up/down 按钮及聚合数量，不显示 “Was this useful?”。
- 当前用户选择使用 shared Toggle/Button selected 状态，不复用风险颜色。
- 已登录：点击未选方向新增或切换；再次点击已选方向取消。
- 未登录：进入 GitHub OAuth；callback 返回后自动完成原投票并恢复对应 Tool Detail。
- 写请求可乐观更新；失败必须回滚并在按钮下显示一行错误。
- 成功不显示 toast，选中态和数量即为反馈。
- 聚合读取失败时显示 unavailable，不把失败伪造成 `0`。

### 详细说明 Dialog

每次成功新增或切换投票后打开 shadcn Dialog：

- `Not now`：关闭。
- `Add details on GitHub`：打开固定仓库 `zation/agent-radar` 的 Tool Feedback Issue Form。
- 取消投票不打开 Dialog。

Issue Form URL 只预填：

- Tool Card ID。
- vote direction。
- release/data version。
- 当前 Tool Card URL。

具体原因在 GitHub 必填，不进入 Agent Radar D1。Worker 和 UI 不直接创建 Issue。

### OAuth 返回定位

ToolsWorkspace 读取一次性 `tool` 与 `feedback` query 参数：

- Tool ID 存在时选择该详情。
- 完成状态同步后使用 `history.replaceState` 移除 OAuth 参数，避免刷新重复表现。
- 未知 Tool ID 回退默认选择，不发起写入。

## GitHub Issue Form

新增 `.github/ISSUE_TEMPLATE/tool-feedback.yml`：

- 固定标签 `tool-feedback`。
- Tool ID、vote、release/data version、Tool URL 为结构化字段。
- 具体原因必填。
- 明确禁止提交 token、secret、私有代码、邮件正文、客户数据或完整 prompt。
- P1 只提供表单入口；P2 才读取、分类和处理 Issue。

## 安全边界

- OAuth 不请求额外 scope。
- 所有 OAuth callback URL 固定为 Worker `/api/auth/github/callback`。
- 写 API 必须同时通过 session、Origin、schema、Tool ID 和频率校验。
- 不记录 access token、OAuth code、Cookie、Client Secret 或 session secret。
- 错误响应不回显 GitHub 原始 token body 或 D1 内部细节。
- Issue 内容始终是不可信输入；P1 不读取或执行 Issue 内容。
- 反馈不能修改风险等级、trust level、critical safety gate 或当前 Rating。

## 失败处理

- GitHub token exchange、GitHub `/user` 或 D1 写入失败：不签发成功 session，不记录投票，返回安全错误页。
- OAuth state 缺失、过期或不匹配：清 cookie，拒绝 callback，提供返回 Tools 链接。
- D1 聚合失败：UI 显示 unavailable。
- Vote mutation 失败：UI 回滚 optimistic state并显示单行错误。
- Session endpoint 失败：UI 退化为匿名只读，不阻断推荐或 Evaluation。
- Logout 失败：保留当前身份显示并允许重试，不在客户端伪装退出。
