# v0.4 UI：shadcn + Tailwind 可维护性重构

> 状态：已批准
>
> 实现提交：无
>
> 当前状态来源：[`docs/15-roadmap.md`](../../15-roadmap.md)、[`docs/14-web-ui.md`](../../14-web-ui.md)
>
> 生命周期：本文件定义 v0.4 Web UI 完成视觉重构后的组件化与可维护性增量。实现完成后补充提交并冻结；Golden Query 源数据与对外文档英文化属于 v0.5。

## 目标

在不改变已批准 Tools / Evaluation 信息架构和浅色 AI 决策终端方向的前提下，把当前大量原生控件与集中式 CSS 重构为以 shadcn primitives 负责组件行为、Tailwind utilities 负责布局和视觉样式的前端结构，并提升全站可读性与交互一致性。

该重构同时服务人类维护者和 coding agent：组件 API、variant、布局与状态边界必须明确，避免巨型 JSX、散落的一次性 class 和难以追踪的全局 CSS 级联。

## 设计原则

### shadcn 负责组件行为

优先复用项目已有组件：

- `Button`
- `Input`
- `Textarea`
- `Select`
- `ToggleGroup`
- 必要时使用 `Dialog` 或其他已安装 primitive

shadcn 组件负责：

- 语义元素和交互 API。
- disabled、pointer、hover、active、focus-visible。
- 键盘与基础无障碍行为。
- 可复用 variant 与尺寸。

页面层不得重复覆盖 shadcn 已提供的 pointer、hover、active 或 focus-visible 样式。只有项目自定义的复合交互组件没有等价 primitive 时，才在该组件的共享 variant 中定义这些状态。

### Tailwind 负责布局与视觉

Tailwind utilities 负责：

- Grid、Flex、容器宽度和响应式断点。
- Padding、gap、margin 和内容密度。
- 字号、字重、line-height 和 tracking。
- 颜色、边框、圆角和表面层级。
- selected、risk、status 等 Agent Radar 领域 variant 的视觉表达。

不得把页面重新实现为默认 shadcn Card 网格。shadcn 是组件与行为基础，不是恢复旧视觉外观。

### CSS 只保留全局职责

`src/ui/styles.css` 仅保留：

- Tailwind 与字体 imports。
- 设计 tokens 和基础 theme 映射。
- `body`、字体渲染等全局基础。
- 无法由 Tailwind 清晰表达的极少量跨组件规则。
- `prefers-reduced-motion` 等全局可访问性规则。

页面布局、组件间距、字号、状态和响应式规则从大段 kebab-case CSS class 迁移到组件内 Tailwind utilities 或共享 component variants。

## 字体与可读性

当前 7–12px 字号过多，必须提升：

- 页面主标题：`text-2xl` 至 `text-3xl`。
- 区域标题：`text-lg` 至 `text-xl`。
- 正文与重要说明：`text-sm` 或 `text-base`，根据密度选择。
- 表单控件、按钮、tool row 主文本：不得小于 `text-sm`。
- 版本、类型、状态、字段标签、分数辅助文案：最低 `text-xs`。
- 不再使用 7–10px 的可见 UI 文字。
- 正文使用舒适 line-height；等宽字体继续只用于版本、状态、类型、字段标签和分数。

桌面端仍保持开发者工具的信息密度，但不能依赖缩小字体压缩内容。

## 文案语言

本次范围内：

- UI chrome、按钮、filter、状态、空状态、错误和静态说明全部使用英文。
- 不新增中文硬编码 UI 文案。
- API 返回的 provider error 等动态文案按既有响应展示，不在浏览器做不可靠翻译。

本次不修改：

- Golden Query 的 `query.task`。
- Golden Query 的 `review_notes`。
- Tool Card 或 Recommendation Result 中来自发布 artifact 的源内容。

Golden Query 源数据、`README.md` 和对外文档英文化已登记为 v0.5，不在 UI 层维护临时翻译副本。

## 组件边界

### App Shell

- 使用 shadcn `Button` variant 构建 Tools / Evaluation 导航。
- Release tag 使用现有图标与 Tailwind 排版。
- 不手写与 Button 重复的交互状态。

### Recommend Composer

- `Textarea`：任务输入。
- `Select`：model 与 risk tolerance。
- `Input`：API key。
- `Button`：Run、Edit、Clear。
- 状态行保持轻量，不恢复旧 Alert 卡片；使用 Tailwind 领域颜色表达 ask-human、no-match 和 error。

### Tools Filters 与 Tool Index

- 搜索使用 `Input`。
- 类型筛选使用 `ToggleGroup` 或共享 shadcn Button variant。
- Tool row 建立独立复合组件，选中态、rank、type 和 score 由明确 props 驱动。
- Tool row 如无法直接由 Button primitive 表达连续索引结构，应复用 Button 行为或定义统一 `tool-row` variant，不在 map 内散落状态 class。

### Evaluation

- Filters 使用同一 ToggleGroup/Button variant，不单独维护一套 hover/focus。
- Query row 使用共享 row component 或 Button variant。
- Health summary 保持中性表面，不能因 shadcn 引入而变回高亮 Card 组。

### Tool Detail

- 保留当前信息顺序和开放式详情表面。
- 不为了组件化给每个 facts、boundary 或 evidence item 套 Card。
- 进度可继续复用 shadcn `Progress`，但颜色与尺寸由 Tailwind variant 统一。

## Agent 可维护性要求

- 每个文件保持单一职责，避免重新把页面合并回巨型 `App.tsx`。
- 重复 class 组合提取到 shadcn variant、共享复合组件或 `cn()` helper。
- 页面组件只组合公开 props，不读取其他组件内部状态。
- 领域状态使用稳定 union types，不用字符串拼接产生未声明 variant。
- 避免 `style={{ ... }}`，评分进度宽度等数据驱动样式除外。
- 新组件必须有明确 props type；纯状态和数据装配继续使用 Node tests。

## 保持不变

- Tools 与 Evaluation 两个一级入口。
- Recommend 与 Tools 的统一工作区。
- Recommend 自适应展开规则。
- 任务候选直接重排 Tool Index。
- Evaluation 的保守完整性判断。
- Desktop 双栏与 Mobile drill-in。
- shadcn 不参与推荐逻辑、评分逻辑或 artifact 数据计算。

## 验证与验收

### 自动验证

```bash
npm run stylelint
npm run lint
npm test
npm run pages:build
```

### 代码验收

- UI 交互控件优先使用已有 shadcn primitives。
- 页面层不重复定义 shadcn 已具备的 pointer、hover、active、focus-visible。
- 大部分页面布局与组件视觉由 Tailwind utilities 表达。
- `styles.css` 显著缩小并只保留全局职责。
- 不出现 7–10px 的可见 UI 文字。
- 不出现新的中文硬编码 UI chrome。
- 不恢复旧 Card 网格或多层嵌套表面。

### 浏览器验收

- Desktop：Tools、Evaluation、filters、tool rows、Recommend 全状态。
- Mobile：导航、Tools drill-in、Evaluation drill-in 与 Back。
- 所有可点击控件具有来自 shadcn/shared variant 的 pointer、hover、active 和 focus-visible。
- 文字在 `1440 × 1000` 与 `390 × 844` 下可读，无裁切或溢出。
- 使用浏览器截图与已批准设计方向对照，确认组件化没有改变信息层级。

- 不修改 README 或对外文档语言。
- 不实现 OAuth、D1 投票或 GitHub Issue Form。
- 不改变推荐、评分或 eval 行为。
- 不恢复 Compare / Review。
- 不引入新的 UI 依赖。
