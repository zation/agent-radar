## 目标

在 v0.3 P1 已建立的可靠 Tool Card、Rating Result 和 reviewed bundle 基线上，为 Web、HTTP API 与 MCP 的动态推荐结果增加不可被 LLM 解除的上下文安全边界、结构化人工确认建议和可执行的关键安全评测门禁。

P2 回答四个问题：

1. 同一个工具用于不同任务时，如何结合工具基础风险和任务上下文判断最终风险？
2. 推荐需要人工确认时，coding agent 和人类用户能否获得明确原因、确认事项与安全默认值？
3. Web、HTTP API、MCP 和 golden eval 是否使用同一条推荐安全链路？
4. 关键安全 case 失败、缺失或未执行时，发布是否会被确定性阻断？

## 当前基线

v0.3 P1 完成后，项目已具备：

- 53 张通过可靠发布门禁的 Tool Cards，以及对应 Rating Results。
- Tool Card 中的权限、信任等级、基础风险、AI decision notes 和字段 provenance。
- Web Recommend 页面、HTTP Recommend API 和 MCP `recommend_tools` 共用的 LLM 推荐引擎。
- 候选归一化、未知工具拒绝、目录候选恢复、`no_reliable_match` 和高风险动作收紧。
- 10 条真实 provider golden queries 及当前 Eval Summary 发布检查。
- 由 Git tag、commit SHA、artifact checksum 和 production evidence 组成的发布溯源链。

当前不足：

- 安全收紧逻辑分散在推荐引擎中，缺少独立、可表驱动测试的上下文安全边界。
- Recommendation Result 没有统一的结构化总体风险、稳定 reason codes、确认事项和安全默认值。
- Web、HTTP API 与 MCP 推荐响应没有在结果内同时暴露 release ID 和 commit SHA。
- 现有 10 条 golden queries 对敏感权限、证据不足和 critical safety 场景覆盖不足。
- 当前发布检查不能明确区分并阻断关键安全 case 的失败、缺失或未执行。

## 范围

### 1. 两层推荐安全评估

新增独立的确定性 `recommendation-safety` 模块。模块输入为：

- `RecommendationQuery`
- 已归一化的候选列表
- 候选对应的 Tool Cards
- 构建期 Rating Results

模块输出统一的 `RecommendationSafetyAssessment`：

```ts
interface RecommendationSafetyAssessment {
  risk_level: RiskLevel;
  reason_codes: SafetyReasonCode[];
  requires_human_approval: boolean;
  approval_reason?: string;
  confirmation_questions: string[];
  safe_defaults: string[];
  maximum_allowed_action: RecommendedAction;
}
```

安全评估分为两层：

1. 工具基础风险：读取 Tool Card 的权限、访问级别、信任等级、安装方式、已知风险和数据流，并复用 Rating Result 的基础风险。
2. 任务上下文风险：结合任务描述、允许权限、风险偏好、候选所需权限和任务影响范围，决定最终风险与最宽松允许动作。

第一版规则必须至少满足：

- `payment`、cloud admin 和 database write 为 `critical`。
- `email`、`secrets`、shell、code execution 和 filesystem write 至少为 `high`。
- browser、network 和 filesystem read 至少为 `medium`。
- unknown permission 不能成为 low risk。
- unknown trust 与 code execution 组合强制 `avoid`。
- high 或 critical 候选不能直接输出 `use`。
- 候选权限超出 `allowed_permissions` 时至少 `ask_human`；明显违反 low risk tolerance 时允许升级为 `avoid` 或 `no_reliable_match`。
- LLM 可以给出更保守的动作，但不能突破 `maximum_allowed_action`。

模块使用稳定 reason codes 和确定性模板生成确认内容。LLM 文本只能用于任务理解、候选选择和候选解释，不能生成或覆盖安全门禁字段。

### 2. Recommendation Result v2

Recommendation Result 直接升级为 `recommendation_result.v2`，Web、HTTP API、MCP 和 golden eval 同步切换。P2 不提供 v1 adapter、版本协商或兼容输出；v1 只保留在 Git 历史中。

```ts
interface RecommendationResult {
  schema_version: "recommendation_result.v2";
  id: string;
  release: {
    release_id: string;
    commit_sha: string;
  };
  query: RecommendationQuery;
  query_understanding: QueryUnderstanding;
  recommended_action: RecommendedAction;
  safety_assessment: RecommendationSafetyAssessment;
  candidates: RecommendationCandidate[];
  rejected_candidates: RejectedCandidate[];
  no_match_reason?: string;
}
```

`release_id` 和 `commit_sha` 必须来自部署时注入的只读版本信息，不能由客户端或 LLM 提供。正式构建缺少任一字段时失败；本地开发使用明确的 `dev` release ID 和可解析的当前 Git SHA，不伪装成 production release。

项目继续保留其他持久化 artifact 和机器接口的 `schema_version`。P2 不执行全局 schema version 删除，也不建设通用多版本管理机制。

### 3. 结构化人工确认建议

当安全评估要求人工确认时，Recommendation Result 必须包含：

- `requires_human_approval: true`
- 稳定的 `reason_codes`
- 面向人的 `approval_reason`
- `confirmation_questions`
- `safe_defaults`

`confirmation_questions` 是 coding agent 在使用工具前应向用户确认的建议问题，不是 Agent Radar 内部审批状态。Web UI 将其呈现为只读的“使用前确认事项”清单，不显示输入框，不收集或保存答案，也不根据答案重新运行推荐。

Agent Radar 不判断用户是否已经授权，不授予第三方权限，不在确认后自动安装或执行工具。

### 4. 24 条 Golden Queries

Golden queries 从 10 条扩展到 24 条，按以下覆盖分组：

- 8 条常见任务与候选排序。
- 8 条权限与人工确认，覆盖 email、filesystem write、shell、code execution、secrets、database write、cloud admin 和 payment。
- 4 条无可靠候选或证据不足，覆盖 unknown permission、unknown trust、高风险偏好冲突和无任务匹配。
- 4 条 critical safety cases，覆盖资金操作、生产数据库写入、cloud admin 和 unknown trust + code execution。

`EvalCase.expected` 扩展为可机器判断的安全断言：

```ts
expected: {
  recommended_action?: RecommendedAction;
  minimum_risk_level?: RiskLevel;
  requires_human_approval?: boolean;
  must_include_reason_codes?: SafetyReasonCode[];
  must_include_confirmation_questions?: boolean;
  must_include_safe_defaults?: boolean;
  must_warn_permissions?: string[];
  should_not_recommend?: string[];
}
```

评测只断言稳定的动作、风险、reason codes、权限提示、候选 ID 和结构化安全字段，不比较自然语言措辞。

### 5. Critical Safety Release Gate

Eval Summary 必须明确记录 critical case：

```json
{
  "critical": {
    "total": 4,
    "passed": 4,
    "failed": 0,
    "release_blocking": false
  },
  "release": {
    "release_id": "all-v0.3.x",
    "commit_sha": "..."
  }
}
```

以下任一情况必须使发布检查返回非零状态，并阻止 reviewed bundle 进入 production deployment：

- 任一 critical case 失败。
- 预期的 critical case 缺失。
- critical case 未执行或只生成 blocked/no-key 占位结果。
- 正式 Release All 中 provider key 缺失、provider 请求失败或响应 schema 无效。
- eval summary 的 release metadata 与当前构建不一致。

普通 golden query 失败仍使完整 P2 发布检查失败；critical 分类额外提供稳定 failure code 和明确的 `release_blocking` 证据。

## 数据流

```text
可靠 Tool Cards + Rating Results + Recommendation Query
  -> LLM 理解任务并选择候选
  -> 候选归一化、未知工具拒绝和必要的目录候选恢复
  -> 确定性 recommendation-safety 评估
  -> 收紧 recommended_action
  -> 生成 reason codes、approval reason、确认事项和 safe defaults
  -> 注入 release ID 与 commit SHA
  -> Recommendation Result v2
  -> Web / HTTP API / MCP / golden eval
```

Golden eval 在 build/release 阶段调用同一条动态推荐链路，但不重新定义 Tool Card 或 Rating Result。P2 不改变 P1 的 Tool Card 采集、normalization、promotion 或 reviewed bundle 数据可信度门禁。

## Web、HTTP API 与 MCP

### Web

Recommend 页面新增：

- 当前 release ID 和短 commit SHA。
- 总体风险等级。
- 人工确认提示。
- 只读的“使用前确认事项”清单。
- 安全默认值列表。

稳定 reason codes 可放在详情或调试信息中，不作为主要用户文案。Web 不实现表单、确认状态、回答持久化、重新推荐或授权按钮。

### HTTP API 与 MCP

HTTP Recommend API 和 MCP `recommend_tools` 返回同一个 `recommendation_result.v2` contract。调用方必须读取结构化安全字段；P2 不提供 v1 兼容分支。

MCP `search_tools`、`get_tool_card`、`explain_rating` 及其他独立响应不因本次 Recommendation Result 升级而修改其 schema contract。

## 异常与保守行为

- LLM 返回未知候选：沿用现有归一化逻辑拒绝并记录原因。
- LLM 返回不安全动作：确定性安全模块收紧，不把模型输出视为安全结论。
- 没有候选可安全使用：返回 `no_reliable_match`，不生成虚假的授权状态。
- 权限或信任未知：采用保守风险，不允许 low-risk `use`。
- release metadata 缺失：本地明确标记为 dev；正式构建失败。
- API key、provider 错误正文和其他 secret：不得写入 Recommendation Result、eval artifact 或可公开日志。

## 测试与验证

### 单元与契约测试

- 安全模块使用表驱动测试覆盖每个权限、访问级别、信任和关键组合规则。
- 推荐引擎测试证明 LLM 不能解除确定性 `ask_human`、`avoid` 或 `no_reliable_match` 下限。
- API/MCP contract tests 验证 v2、release metadata 和结构化确认字段。
- Web view tests 验证风险提示、只读确认清单和 safe defaults。
- release metadata tests 验证 production 缺失失败和 dev 明确标记。

### Golden Eval 与发布检查

- 24 条 golden queries 全部执行。
- 4 条 critical safety cases 全部存在并通过。
- critical case 失败、缺失或未执行均使检查命令返回非零状态。
- 正式 Release All 使用真实 provider，不接受无 key 或 blocked 占位结果。
- 全量 tests、lint、stylelint 和 production build 通过。

## 非目标

P2 不包含：

- Eval Diff、历史 artifact 下载或依赖上一成功 build 的比较机制；Eval Diff 移入 Backlog。
- Prompt、provider routing 或推荐规则的独立版本号、注册表或回放机制。
- Recommendation Result v1 兼容输出、adapter 或版本协商。
- 全局移除持久化 artifact 的 `schema_version`。
- Web 审批表单、确认状态、答案持久化或企业审批流。
- 自动授权、安装或执行推荐工具。
- 修改 P1 Tool Card 采集、normalization、promotion 和数据可信度主链路。
- Provider 配置 UI、浏览器 direct-to-provider 或 proxy 模式重构。
- GitHub OAuth、D1 投票和反馈评级接入；这些属于 v0.4。

## 完成标准

- 所有推荐入口统一输出 `recommendation_result.v2`。
- Web、HTTP API、MCP 和 golden eval 使用同一确定性安全模块。
- Web、HTTP API 和 MCP 返回一致的安全字段、release ID 和 commit SHA。
- high/critical、unknown permission 和 unknown trust + code execution 场景满足明确的安全下限。
- 所有 `ask_human` 结果包含结构化原因、确认事项和安全默认值。
- Web 只展示确认清单，不收集或保存答案。
- 24 条 golden queries 全部执行，4 条 critical safety cases 全部通过。
- critical case 失败、缺失或未执行会阻断 reviewed bundle 和发布。
- 全量 tests、lint、stylelint 和 production build 通过。
