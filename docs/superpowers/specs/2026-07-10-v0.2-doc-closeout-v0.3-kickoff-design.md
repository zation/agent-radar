## 文档处理规则

### Roadmap 与 Superpowers 文档

`docs/15-roadmap.md` 和 `docs/superpowers/**` 用于记录计划演进，允许保留已经被替代或取消的历史文本。

- 废弃内容使用 Markdown 删除线 `~~旧内容~~`。
- 删除线后必须紧跟当前实际情况、替代方案或完成证据。
- 已完成但仍以未来时态表达的任务改为完成状态。
- 已执行的计划步骤可改为 `[x]`；若实际执行方式与原计划不同，保留原步骤并用删除线标明，再补充实际执行方式。
- 不得只加“已归档”标记而保留会误导当前状态的正文。

### 其他权威文档

README、产品简报、需求、架构、采集、评测、部署和 Web UI 等权威文档直接改成当前事实。

- 不使用删除线保留旧状态。
- 不并列保留互相冲突的“当前 baseline”。
- 历史版本只在确有审计价值时作为历史基线说明，不得称为当前状态。
- “最小实现”和“完整实现”必须明确区分，避免把已有基础能力写成尚未实现，或把后续增强写成当前完成。

## v0.3 Kickoff 内容

### P1：数据与可信度

- 增加高价值来源和 Tool Cards，目标范围为 50–150 张。
- 完善跨来源字段冲突、alias/package key、deduper、normalizer 和发布准入。
- 提升字段 provenance、URL 可达性和 reviewed bundle 审核证据质量。
- 增加 eval diff，比较 prompt/provider/rule 变化前后的推荐动作、候选、风险提示和失败分类。

### P2：Provider 能力

- 将 `provider_registry.json` 接入完整 Provider 配置 UI。
- 明确浏览器运行时读取配置的方式。
- 明确 browser direct-to-provider 与 Worker proxy 的安全边界。
- 为 prompt、provider routing 和评测结果增加可追溯版本号。

### 后续 v0.3 能力

- 安全风险评分和高风险评测发布阻断。
- 受限的用户反馈记录、汇总和写入口。
- 自动 Review Summary。
- 数据质量 dashboard 或报告。
- parser failure、data quality、recommendation misrank 和 safety eval failure 等自迭代任务。
- Golden queries 扩展到 20–40 条。

### v0.3 明确不做

- 自动信任新来源。
- 自动安装或执行第三方工具。
- 完全自动修复所有误判。
- 企业级审批流、账号系统、SSO 或商业 marketplace。
- 新的付费服务或长期付费基础设施。
