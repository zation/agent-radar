# v0.2 收尾设计

## 目标

完成 Agent Radar v0.2，使代码、生成产物、文档、GitHub production 审批、部署证据和线上 Worker 对系统状态的描述完全一致，并形成一次内部一致、可复现的发布。

目标版本为 `all-v0.2.5`。它将在已经成功部署的 `all-v0.2.4` 基础上，纳入已经提交的审核流程修正。

## 已确认的产品边界

v0.2 包含：

- 通过采集和 promotion gates 发布 20-50 张 Tool Cards。
- 10-20 条 golden queries，所有 critical cases 使用真实 LLM provider 运行并通过。
- 只读 Web UI、HTTP API 和 MCP JSON-RPC endpoint，统一部署在一个 Cloudflare Worker 中。
- 将脚本、规则和 LLM 生成的审核证据持久化到 reviewed bundle。
- 只在 GitHub `production` environment gate 进行一次人工发布确认。
- 持久化不可变的 production evidence，将已批准 run、源码 SHA、reviewed bundle、checksums、已部署 Worker 和 MCP smoke result 关联起来。

v0.2 不包含：

- 更完整的 Provider 配置 UI，或浏览器运行时读取 `provider_registry.json`。
- 浏览器 direct-to-provider 模式。
- 逐条人工 approval 表单或 JSON review record 生成器。
- 用户反馈写入、账号管理、定时采集、启用 D1 serving cache 或自动执行高风险动作。

Provider UI 和 direct/proxy 决策移到 v0.3/P2，不阻塞 v0.2。

## 当前证据

`all-v0.2.4` GitHub Actions run 已证明功能 baseline：

- 发布 29 张 Tool Cards。
- 真实 provider golden queries 10/10 通过。
- promotion candidates 29/29 通过 promotion check。
- 经过 GitHub `production` environment gate 后成功完成生产部署。
- 已部署 MCP 在 `https://agent-radar.zation1.workers.dev/api/mcp` 完成 4/4 smoke checks。

剩余工作是质量和一致性收尾，不再新增产品功能范围。

## 设计

### 1. 质量门禁清理

修复 `src/ingestion/normalizer.ts`、`tests/pipeline.test.ts` 和 `tests/worker.test.ts` 中现有的 11 个 ESLint errors，不改变运行时行为。删除只服务于已移除的人工 review record JSON preview 的 CSS。

最终本地门禁为：

```text
npm test
npm run lint
npm run stylelint
npm run pages:build
git diff --check
```

真实 provider release build 继续由 GitHub Actions 执行，因为 provider key 只保存在 GitHub secret 中。

### 2. 审核语义与生成产物清理

正常发布路径为：

```text
ingestion -> validation/dedup -> auto review -> release admission
  -> promotion check -> reviewed bundle -> GitHub production approval
```

`approval_record.v1` 仅保留为 break-glass `approval_override`。任何 UI 或生成的 request template 都不应暗示维护者需要逐条批准 draft。

需要在发布时关注的 discovery candidates 使用 production gate 语义，不再使用 `pending_manual_review`。对应 artifact 字段和展示文案应表达“等待 production gate review 或 intervention”。这不影响推荐阶段的 `requires_human_approval`；高风险工具使用仍然必须要求人工确认。

Pipeline 写入新 artifacts 前必须删除废弃的 `data/approval_requests` 输出。`public/data` 保持为不纳入 Git 的生成目录。回归测试必须证明：即使输出目录中预先存在旧 approval request 文件，重新构建也不会让它们进入新 bundle，并会生成 `data/intervention_requests`。

### 3. 文档一致性

更新 README 以及产品、架构、来源、采集、评测、部署、UI 和路线图等权威文档，使其一致表达：

- Web、artifacts、HTTP API 和 MCP 由一个启用 Static Assets 的 Cloudflare Worker 承载；旧 Pages workflows 属于历史方案，不是当前发布架构。
- MCP JSON-RPC endpoint 和部署后 smoke 已实现并部署。
- 受控的公开 GitHub topic metadata 和 npm package metadata 已作为 enabled sources，仍必须经过 validation、auto review 和 promotion gates。
- Review records 不是正常持久化机制；正常机制是 reviewed bundle evidence 加 GitHub production approval。
- Provider 配置 UI 延期到 v0.2 之后。
- `all-v0.2.5` 在全部验证成功后作为 v0.2 收尾版本。

只有以下语境保留人工确认：高风险工具执行、安全边界变化、破坏性操作，以及 GitHub production release gate。

### 4. Production Release Evidence

新增一个小型 production evidence builder，输出版本化 JSON。输入包括：

- Reviewed bundle 中的 `dist-pages/artifact-manifest.json`。
- 已部署 Worker smoke 生成的 `mcp-smoke-result.json`。
- GitHub run id、SHA、ref/tag、repository 和 production deployment id。
- 从 Wrangler 输出解析出的 Worker base URL。

它生成 `production-release-evidence.json`，包含：

- `schema_version`。
- GitHub repository、run id、SHA 和 release tag。
- GitHub production deployment id。
- Reviewed bundle artifact name。
- Artifact manifest SHA-256。
- 从 manifest checksum map 读取的 D1 seed SHA-256。
- Worker base URL 和 MCP endpoint。
- MCP smoke pass/fail summary。
- 生成时间。

出现以下情况时 builder 必须失败：deployment identifier 缺失、D1 seed checksum 不一致、manifest `git_sha` 与 `GITHUB_SHA` 不一致、smoke result 失败，或 smoke endpoint 不属于当前部署的 Worker base URL。

Release workflow 在 smoke 后使用 `gh api` 查询匹配的 production deployment，运行 builder，把精简 evidence summary 追加到 `GITHUB_STEP_SUMMARY`，并上传 JSON 和 smoke artifact。该过程只记录证据，不取代或自动完成 production 人工审批。

### 5. 发布流程

本地验证通过后：

1. 提交实现和文档变更。
2. 使用 Git/`gh` 兼容的仓库认证推送 `main`。
3. 创建并推送 annotated tag `all-v0.2.5`。
4. 使用 `gh` 监控 `Release All` workflow。
5. 维护者在 GitHub 手动批准 `production` environment job。
6. 验证 workflow、production evidence artifact、线上 `/api/version`、线上 data manifest 和线上 MCP endpoint。
7. 确认线上 release id 为 `all-v0.2.5`、Tool Card 数量位于 20-50、golden eval 至少 10/10 且 MCP smoke 全部通过。

v0.2 不要求创建 GitHub Release 对象。不可变 tag、Actions run、上传的 evidence artifacts、GitHub deployment record 和线上 version endpoint 共同构成发布记录。

## 错误处理与回滚

- 任一本地质量门禁失败都阻止完成提交。
- 真实 provider eval、promotion check、Worker dry run、deployment、evidence validation 或 MCP smoke 任一失败，都阻止发布被标记为成功。
- 无法取得匹配的 GitHub production deployment id 时，evidence generation 失败，并使 production job 失败。
- 如果 `all-v0.2.5` 部署失败，`all-v0.2.4` 仍是最后一个已验证的 production 版本。恢复时使用 Cloudflare Worker rollback，或重新部署之前的 immutable reviewed bundle，不重新构建旧数据。

## 非目标与风险

本次收尾不扩大来源覆盖、不修改评分权重、不增加推荐模型、不启用 D1 serving，也不重新设计 UI。主要发布风险是 Cloudflare deploy 后的 workflow evidence collection；该逻辑位于部署后独立执行，并通过纯 builder 测试，使失败能够被明确识别。Production gate 仍严格保持为 GitHub 人工审批。
