# v0.2 Closeout Design

## Goal

Finish Agent Radar v0.2 with one internally consistent, reproducible release whose code, generated artifacts, documentation, GitHub production approval, deployment evidence, and live Worker all describe the same system.

The target release is `all-v0.2.5`. It supersedes the successful `all-v0.2.4` deployment after carrying forward the already committed review-flow correction.

## Confirmed Product Boundary

v0.2 includes:

- 20-50 released Tool Cards generated through the ingestion and promotion gates.
- 10-20 golden queries with all critical cases passing against a real LLM provider.
- A read-only Web UI, HTTP API, and MCP JSON-RPC endpoint deployed in one Cloudflare Worker.
- Script/rule/LLM-backed review evidence persisted in the reviewed bundle.
- One human release decision at the GitHub `production` environment gate.
- Immutable production evidence tying the approved run, source SHA, reviewed bundle, checksums, deployed Worker, and MCP smoke result together.

v0.2 does not include:

- A richer Provider configuration UI or runtime loading of `provider_registry.json` in the browser.
- Browser direct-to-provider mode.
- Per-item manual approval forms or JSON review-record generation.
- User feedback writes, account management, scheduled crawling, D1 serving-cache activation, or automatic high-risk actions.

The Provider UI and direct/proxy decision move to v0.3/P2 and do not block v0.2.

## Current Evidence

The `all-v0.2.4` GitHub Actions run proves the functional baseline:

- 29 released Tool Cards.
- 10/10 real-provider golden queries passed.
- 29/29 promotion candidates passed the promotion check.
- Production deployment completed after the GitHub `production` environment gate.
- Deployed MCP smoke passed 4/4 checks at `https://agent-radar.zation1.workers.dev/api/mcp`.

The remaining work is closeout quality and consistency, not a new product feature set.

## Design

### 1. Quality Gate Cleanup

Fix the 11 existing ESLint errors in `src/ingestion/normalizer.ts`, `tests/pipeline.test.ts`, and `tests/worker.test.ts` without changing runtime behavior. Remove CSS that only supported the deleted manual review-record JSON preview.

The final local gate is:

```text
npm test
npm run lint
npm run stylelint
npm run pages:build
git diff --check
```

The real-provider release build remains a GitHub Actions responsibility because the provider key is stored only as a GitHub secret.

### 2. Review Semantics and Generated Artifact Cleanup

The normal release path is:

```text
ingestion -> validation/dedup -> auto review -> release admission
  -> promotion check -> reviewed bundle -> GitHub production approval
```

`approval_record.v1` remains only as a break-glass `approval_override`. No UI or generated request template should suggest that maintainers normally approve drafts one by one.

Discovery candidates that need attention at release time use production-gate language rather than `pending_manual_review`. Their artifact fields and rendered copy should describe a pending production-gate review or intervention. This change does not affect recommendation-time `requires_human_approval`, which remains required for high-risk tool use.

The pipeline must delete the deprecated `data/approval_requests` output before writing new artifacts. The repository's tracked generated data must also remove that directory and include `data/intervention_requests` instead. A regression test proves that rebuilding from a directory containing legacy approval-request files cannot leak them into a new bundle.

### 3. Documentation Consistency

Update README and the authoritative product, architecture, source, ingestion, evaluation, deployment, UI, and roadmap documents so they consistently state:

- Web, artifacts, HTTP API, and MCP run in one Cloudflare Worker with Static Assets; old Pages workflows are historical, not the current release architecture.
- The MCP JSON-RPC endpoint and deployment smoke are implemented and deployed.
- Controlled public GitHub topic metadata and npm package metadata are enabled sources that still pass validation, auto review, and promotion gates.
- Review records are not the normal persistence mechanism; reviewed-bundle evidence plus GitHub production approval is.
- Provider configuration UI is deferred beyond v0.2.
- `all-v0.2.5` is the closeout release after successful verification.

References to human confirmation remain only where they mean high-risk tool execution, security-boundary changes, destructive operations, or the GitHub production release gate.

### 4. Production Release Evidence

Add a small production-evidence builder with a versioned JSON output. It consumes:

- `dist-pages/artifact-manifest.json` from the reviewed bundle.
- `mcp-smoke-result.json` from the deployed Worker smoke test.
- GitHub run id, SHA, ref/tag, repository, and production deployment id.
- The deployed Worker base URL parsed from Wrangler output.

It produces `production-release-evidence.json` containing:

- `schema_version`.
- GitHub repository, run id, SHA, and release tag.
- GitHub production deployment id.
- Reviewed bundle artifact name.
- Artifact manifest SHA-256.
- D1 seed SHA-256 copied from the manifest checksum map.
- Worker base URL and MCP endpoint.
- MCP smoke pass/fail summary.
- Generation timestamp.

The builder rejects missing deployment identifiers, a D1 seed checksum mismatch, a manifest `git_sha` that differs from `GITHUB_SHA`, a failed smoke result, or a smoke endpoint outside the deployed Worker base URL.

The release workflow queries the matching production deployment with `gh api`, runs the builder after smoke, appends a compact evidence summary to `GITHUB_STEP_SUMMARY`, and uploads the JSON alongside the smoke artifact. This records evidence; it does not replace or automate the human production approval.

### 5. Release Procedure

After local verification:

1. Commit the implementation and documentation changes.
2. Push `main` using Git/`gh`-compatible repository authentication.
3. Create and push annotated tag `all-v0.2.5`.
4. Monitor the `Release All` workflow with `gh`.
5. The maintainer manually approves the `production` environment job on GitHub.
6. Verify the workflow, production evidence artifact, live `/api/version`, live data manifest, and live MCP endpoint.
7. Confirm the live release id is `all-v0.2.5`, the released Tool Card count is within 20-50, golden eval is 10/10 or better, and MCP smoke is fully passing.

No GitHub Release object is required for v0.2; the immutable tag, Actions run, uploaded evidence artifacts, GitHub deployment record, and live version endpoint are the release record.

## Error Handling and Rollback

- Any local quality-gate failure blocks commit completion.
- Any real-provider eval, promotion check, Worker dry run, deployment, evidence validation, or MCP smoke failure blocks a successful release.
- Failure to obtain the matching GitHub production deployment id blocks evidence generation and therefore fails the production job.
- A failed `all-v0.2.5` deployment leaves `all-v0.2.4` as the last verified production version. Recovery uses Cloudflare Worker rollback or redeploys the prior immutable reviewed bundle; it does not rebuild old data.

## Verification Matrix

| Requirement | Evidence |
| --- | --- |
| 20-50 Tool Cards | Reviewed bundle `data/tool_cards.jsonl` count and live artifact count |
| 10-20 golden queries | Reviewed bundle `data/golden_queries.json` and `eval_summary.json` |
| Critical eval passes | Release build gate and artifact manifest eval summary |
| Stable read-only MCP | Contract tests plus deployed MCP smoke result |
| Automated review persistence | Manifest, ingestion review, auto-review, release-admission, promotion, and intervention artifacts |
| Human gate only at production | GitHub environment protection plus workflow job dependency |
| No deprecated approval requests | Pipeline regression test and reviewed-bundle path scan |
| Deployment traceability | `production-release-evidence.json` plus GitHub deployment record |
| Documentation consistency | Targeted terminology search and roadmap acceptance audit |
| Repository quality | Test, ESLint, Stylelint, Vite build, and whitespace checks |

## Non-Goals and Risk

This closeout does not broaden source coverage, change rating weights, add recommendation models, activate D1 serving, or redesign the UI. The main release risk is workflow evidence collection after Cloudflare deploy; it is isolated after deployment and tested as a pure builder so failures are explicit. The production gate remains a manual GitHub approval exactly as required.
