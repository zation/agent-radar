# v0.9 P3: Release Identity Convergence Gate Design

- Status: Approved
- Implementation commits: None
- Current status source: [docs/15-roadmap.md](../../15-roadmap.md)

## Background

The `all-v0.9.0` and `all-v0.9.1` releases exposed an edge-propagation race: immediately after Wrangler reported a successful deployment, `/api/version` and MCP `initialize.serverInfo.version` could still come from the previous Worker version. The existing seven-check smoke accepted any non-empty server version, while its `release_id` and `commit_sha` fields merely copied expected workflow inputs. A functional PASS therefore did not prove that the immutable tag, deployed Worker, MCP server, and Registry publication had converged on one identity.

## Goals

1. Poll the deployed Worker's `/api/version` with bounded retries until both `release_id` and `commit_sha` exactly match the immutable release tag and source SHA.
2. Derive the expected MCP server version from the same canonical `all-v*` tag parser used by Registry metadata.
3. Require MCP `initialize.serverInfo.version` to equal that expected version.
4. Reuse one identity-convergence implementation in Release All and Publish MCP Registry.
5. Persist expected and actual release ID, commit SHA, MCP server version, attempt count, endpoint, and timestamps in production evidence.
6. Block release evidence creation and Registry publication on mismatch, malformed responses, request exhaustion, or timeout.

## Non-Goals

- Changing HTTP, MCP tool, Skill dataset, or Registry metadata public contracts.
- Replacing Cloudflare deployment or introducing a new service.
- Treating retries as proof of global edge convergence beyond the checked production endpoint.
- Retrying all seven functional MCP checks; only release identity receives an explicit convergence phase.

## Identity Convergence Contract

`release_identity_convergence.v1` records the checked `/api/version` URL, expected and final actual release ID and commit SHA, total attempts, start/end timestamps, and `converged: true`. The builder returns evidence only after exact identity equality. Exhaustion throws and produces no successful evidence artifact.

The runner uses a fixed attempt count and interval supplied by trusted workflow configuration. Each request is uncached, has an abort timeout, accepts JSON only, and does not include credentials. Error messages report status or identity mismatch without echoing response bodies.

## MCP Smoke Contract

`mcp_smoke_result.v3` replaces copied top-level release labels with an `identity` object containing `expected_server_version` and `actual_server_version`. The initialize check requires the canonical Agent Radar server name and exact expected version. A missing expected version is a configuration error rather than a permissive smoke.

The other six functional checks remain unchanged, preserving the exact seven-check contract.

## Production and Registry Evidence

`production_release_evidence.v2` binds the reviewed bundle, GitHub deployment, convergence artifact, and v3 MCP smoke. Its identity section records expected and actual release ID, commit SHA, server version, convergence attempt count, and convergence timestamps. Evidence construction rejects any disagreement among GitHub tag/SHA, convergence result, Worker origin, MCP endpoint, and smoke result.

Publish MCP Registry accepts only v2 production evidence and v3 source smoke, reruns the same convergence CLI against the metadata-bound Worker origin, then runs a fresh version-aware smoke before OIDC publication. The fresh convergence and smoke artifacts are uploaded with Registry publication evidence.

## Acceptance

1. Unit tests cover immediate convergence, retry-then-converge, timeout/exhaustion, malformed response, release mismatch, and SHA mismatch.
2. MCP smoke tests prove exact version success and previous-version failure.
3. Production evidence tests prove expected/actual identity binding and reject tampered convergence or smoke fields.
4. Workflow tests prove convergence precedes smoke and Registry publication in both workflows and that expected versions are derived from immutable tags.
5. `npm test`, lint, stylelint, Pages build, public-language check, and diff checks pass.

## Risks

- A single successful endpoint check cannot prove every CDN edge has converged; it proves the production endpoint observed by the release runner has the intended immutable identity.
- Too-short retry bounds can create false release failures during legitimate propagation. Bounds must remain explicit and conservative.
- Evidence schema upgrades require synchronized validation in Registry publication and tests; mixed v1/v2 or v2/v3 artifacts are intentionally rejected.
