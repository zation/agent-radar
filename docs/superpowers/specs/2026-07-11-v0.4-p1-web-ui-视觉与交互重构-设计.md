# v0.4 P1：Web UI 视觉与交互重构

> 状态：已批准
>
> 实现提交：无
>
> 当前状态来源：[`docs/15-roadmap.md`](../../15-roadmap.md)、[`docs/14-web-ui.md`](../../14-web-ui.md)
>
> 生命周期：本文件定义 v0.4 P1 的 Web UI 视觉与交互重构。实现完成后补充实现提交并冻结；后续视觉迭代创建范围更小的新 Spec。

## 目标

把当前基于通用 shadcn 组件外观的多页面 Web UI，重构为面向开发者的浅色 AI 决策终端。新界面必须让用户在一条连续路径中完成任务描述、候选排序、工具浏览、证据审查和反馈，同时把推荐评测从内部术语转化为可理解的公开透明度页面。

视觉优化不能改变 Agent Radar 的产品边界：系统仍是只读工具评级与推荐知识库，不安装、授权或执行第三方工具。v0.4 P1 新增的反馈写操作继续受 GitHub OAuth、D1 唯一投票和 GitHub Issue Form 边界约束。

## 已批准的设计方向

### 产品气质

采用“浅色可信情报终端”，而不是深色监控台、营销型 SaaS Dashboard 或默认组件库后台：

- 未来感来自信息结构、等宽元数据、精确状态和开发者工具式密度。
- 白灰绿低饱和底色服务长时间阅读和证据审查。
- 不使用装饰渐变、霓虹、玻璃拟态、营销插画或大面积阴影。
- 页面保持适度留白，但桌面首屏仍能看到任务、候选、推荐理由和核心证据。
- 列表和详情优先使用排版、分隔线与连续表面；卡片只用于真正独立的语义块。

### 一级导航

顶部只保留两个入口：

- `Tools`
- `Evaluation`

不为 `Compare` 和 `Review` 预留导航或页面结构。本次重构不实现它们；后续可以独立决定删除旧实现还是迁移其少量能力。

`Golden Queries` 不作为一级导航名。它是评测方法术语，在 `Evaluation` 页面正文中解释。

## Tools 页面

### 连续决策路径

页面从上到下固定为：

1. 顶部导航和 release tag。
2. Recommend 命令区。
3. Tools 标题、搜索和类型筛选。
4. 桌面端工具列表与 Tool 详情双栏。

Recommend 不再是独立页面。用户提交任务后，结果直接重排下面的 Tools 列表；不在输入区下方新增结果卡片或结果说明面板。

没有当前任务时，列表展示完整已审核工具索引。存在成功推荐时，列表切换为任务相关候选顺序，并显示候选 rank、工具评分和推荐等级。清除任务后恢复完整索引。

搜索和类型筛选作用于当前列表：存在推荐任务时筛选已排序候选；没有任务时筛选完整索引。

### Recommend 展开规则

Recommend 使用自适应展开：

- 首次访问或没有当前任务：默认展开。
- 推荐成功：自动折叠成单行任务摘要。
- 加载中：保持展开。
- `ask_human`、`no_reliable_match` 或请求错误：保持展开。
- 清除任务：清除结果并恢复展开。
- 用户可随时手动展开或折叠；折叠不能取消当前任务。

成功折叠行只显示：

- 绿色成功图标。
- 当前任务摘要。
- `N candidates ranked`。
- `Edit` 展开控件。

成功状态不在命令区下方显示额外解释。

展开状态包含任务输入、model、risk tolerance、API key 和提交按钮。API key 继续明确标注仅用于本次请求且不持久化。

### Recommend 状态

- Loading：保持展开；显示正在分析工具适配、风险和证据的具体进度文案。
- Success：自动折叠；只显示成功图标、任务摘要和候选数。
- Ask Human：保持展开；在输入框正下方显示单行 amber 信息。它是安全决策状态，不使用红色错误语义，不显示下一步按钮。
- No Reliable Match：保持展开；在输入框正下方显示单行中性信息，不伪造候选，不显示下一步按钮。
- Provider 或 Request Error：保持展开并保留输入；在输入框正下方显示单行红色错误信息，允许通过原提交按钮重试。

状态信息必须位于命令区内部，不能在 Recommend 下方生成独立说明面板。

### 工具列表

- 使用连续索引，不把每条工具记录渲染成独立浮动卡片。
- 选中态使用浅色背景与左侧 trust-color rail。
- 工具类型使用小型、稳定的类型色点；颜色不能承担推荐或风险语义。
- 每行优先显示名称、类型、推荐等级、rank 和 score。
- hover、focus 和 selected 必须可区分。

### Tool 详情层级

详情按以下顺序展示：

1. 工具类型、名称、摘要、总分和风险。
2. 总分下方的 thumbs up/down 图标按钮与聚合计数。
3. 当前任务下的 `Why Radar recommends this`；没有任务时显示稳定的 rating explanation，而不伪造任务理由。
4. Decision、Evidence、Maintenance 和 Integration 四个快速判断字段。
5. 无标题的 `Good for` 与 `Not for` 两列摘要。
6. `Decision signals` 分项评分。
7. `Evidence & access`：主要来源、权限、安全说明、最后验证时间和 release tag。

`Good for / Not for` 的优先级高于详细评分与证据。它们不放入名为 `Usage boundaries` 的外层标题或卡片。

分项评分必须保留数值、可视进度和 release 归属说明。总分不能替代风险、证据、权限或不适用边界。

### 反馈交互

总分下方只显示语义明确的 thumbs up/down 图标与聚合计数，不显示 `Was this useful?` 或常驻 GitHub 反馈文案。

图标按钮必须提供：

- hover/focus tooltip。
- `Upvote this tool record` / `Downvote this tool record` 无障碍名称。
- 当前用户投票的浅色背景和边框选中态。
- GitHub 未登录时触发最小登录流程。
- 同一用户对同一 Tool Card 的赞、踩、切换和取消保持幂等。

完成一次赞或踩后，显示轻量弹窗：询问用户是否愿意前往 GitHub 补充字段错误、文档过期、安装失败或权限异常等详细原因。弹窗只包含 `Not now` 与 `Continue to GitHub`；页面本身不存储自由文本，也不直接创建 GitHub Issue。

## Evaluation 页面

### 命名与定位

一级导航使用 `Evaluation`，页面标题使用 `Recommendation Evaluation`。正文解释当前评测集由固定、可回放的 golden queries 组成，并说明它们用于防止推荐质量、安全边界和无可靠结果场景随版本退化。

### 页面结构

页面包含：

- Pass rate、critical cases 和 last run 质量摘要。
- 按 critical、expected action 或场景类型筛选。
- Golden query 列表。
- 选中 query 的目的、输入任务、预期结果、实际结果、风险断言、top candidate、release、证据状态、保护边界和最近变化。
- 明确的“为什么需要这条 Query”说明。
- 明确的 release-blocking 失败条件。

### 状态语义

三个质量指标使用统一的中性容器，不使用整块绿色背景、底部绿色状态线或其他类似 filter 选中的样式。

- 全部通过：绿色 check icon + `ALL PASSED`。
- 存在失败：红色 warning icon + 失败数量。
- Last run：主值显示相对或绝对运行时间；其下的 release/version 使用 tag icon。
- Clock icon 只表示时间。
- Tag icon 只表示 release/version。
- Check/warning icon 只表示评测状态。

筛选选中态只在指标下方的 filter 行表达，不与状态语义复用。

## 响应式设计

### Desktop

- Tools 使用工具列表与详情双栏。
- Recommend 展开时仍保证列表和详情在首屏下半部可见；成功折叠后优先扩大工具工作区。
- Evaluation 使用 query 列表与详情双栏。

### Mobile

Tools 使用 drill-in 导航：

- 默认显示 Recommend、搜索、筛选和工具列表。
- 点击工具进入独立 Tool 详情视图。
- 返回列表时保留当前任务、推荐排序、搜索、类型筛选和滚动位置。
- 不在列表内使用长 accordion 展开完整详情。

Evaluation 使用同一模型：query 列表进入 query 详情，返回时保留筛选和滚动位置。

移动端导航不隐藏 Recommend 主路径；成功状态仍折叠为紧凑任务摘要。

## 设计系统

### 颜色

建议起始 token：

- Canvas：`#edf2f0`
- Surface：`#f9fbfa`
- Ink：`#17302a`
- Trust：`#087d69`
- Caution：`#d2932a`
- Error：`#c95648`

实现时允许为对比度、hover、focus 和 disabled 补充派生色，但不得改变浅色低饱和基调。

### 字体

- Geist Variable 用于标题、正文和控件。
- 系统等宽字体用于版本、状态、类型、字段标签和分数。
- 等宽字体不用于长说明正文。
- 标题使用紧凑 tracking；正文使用更宽松 line-height。

### 圆角与容器

- 输入、按钮、filter 和小型控件：约 `4px`。
- 命令区、弹窗和独立功能容器：约 `5–8px`。
- 不使用大号圆角、胶囊化页面框架或多层嵌套卡片。
- 阴影只用于弹窗或必要的浮层；常规列表和详情依靠边框与分隔线。

### 三类视觉语义

- 状态：图标 + 文案 +语义色。
- 选中：背景、边框或左侧 rail。
- 强调：字号、字重和排版位置。

三者不得复用同一种主要视觉处理，避免把成功状态误解为当前选中。

### 图标

- 图标风格统一为简洁线性图标，保持相同 stroke weight 和 optical size。
- Check/warning 表示状态。
- Clock 表示时间。
- Tag 表示 release/version。
- Thumbs up/down 表示反馈。
- 展开、返回、编辑和外链使用明确的方向或动作图标。
- 不能使用纯文本符号替代生产图标。

### 单一视觉锚点

- Tools 的强视觉锚点是 Recommend 命令区。
- Evaluation 的强视觉锚点是质量摘要，但摘要保持中性容器。
- 同一页面不得出现多个争夺注意力的高饱和面板。

## 组件与状态边界

实现应保留清晰组件所有权，至少拆分为：

- App shell 和 top navigation。
- Recommend composer、collapsed summary 和 inline status。
- Tool filters、tool list 和 tool row。
- Tool detail header、feedback controls、decision summary、boundaries、signals 和 evidence。
- Evaluation summary、filters、query list 和 query detail。
- Mobile list/detail navigation state。
- Feedback confirmation dialog。

允许继续复用无障碍行为可靠的 shadcn/Base UI primitive，但最终视觉不能由其默认 Card、Tabs、Badge、Alert 或 Toggle 外观决定。重复元素使用共享 token 和明确 variant，不使用散落的一次性样式。

## 数据与交互边界

- Tool Card、Rating Result、Recommendation Result 和 Eval Summary 继续来自现有共享 artifacts/API，不在 UI 重算。
- Recommend 继续通过同一 Worker 后端代理调用 provider，API key 不持久化。
- `ask_human` 不授权、不保存确认答案、不执行工具。
- Evaluation 只读，不在浏览器端重跑 golden queries。
- 反馈自由文本只存在于用户主动打开的 GitHub Issue Form。
- Compare 和 Review 不属于本次新设计，不得阻塞 Tools 与 Evaluation 的实现。

## 异常与空状态

- Artifacts 加载失败：显示明确错误和 release/data 上下文，不渲染空壳工作区。
- 搜索无结果：保留搜索和筛选，显示中性空状态。
- 推荐无候选：使用 Recommend 输入区内的单行 `no_reliable_match` 信息。
- 推荐请求失败：保留任务、model、risk 和 API key 输入状态，仅显示脱敏错误。
- Evaluation 缺失或 blocked：质量摘要显示红色警告，不能伪装成 0/0 通过。
- GitHub 登录或投票失败：保留当前页面与选择，使用靠近反馈按钮的紧凑错误反馈；不能修改聚合数造成假成功。

## 可访问性

- 所有交互必须支持键盘和可见 focus。
- 颜色不能单独承担状态、风险、选中或类型含义。
- 图标按钮必须有可访问名称和 tooltip。
- 文字与背景满足 WCAG AA 对比度。
- 动画尊重 `prefers-reduced-motion`。
- 移动端返回行为和浏览器历史必须可预测。

## 验证与验收

### 自动验证

实现后至少运行：

```bash
npm run stylelint
npm run lint
npm test
npm run pages:build
```

涉及推荐展示、反馈数据装配或 Evaluation artifacts 时，按改动范围追加：

```bash
npm run eval
npm run pipeline
```

### 浏览器验收

- Desktop Tools：首次展开、成功折叠、手动编辑、清除任务、搜索、筛选、候选选择和详情下钻。
- Recommend：loading、success、ask_human、no_reliable_match 和 provider error。
- Tool feedback：未登录、已登录、赞、踩、切换、取消、失败和 GitHub 提示弹窗。
- Evaluation：全部通过、普通失败、critical failure、筛选和 query 详情。
- Mobile：Tools 列表到详情、返回状态恢复；Evaluation 列表到详情、返回状态恢复。
- 键盘、focus、tooltip、dialog focus trap、reduced motion 和基础屏幕阅读器名称。

### 视觉验收

- 与已批准视觉工作区中的 Tools、Recommend states、Tool detail、Evaluation、Mobile 和 Design System 画面逐项对照。
- 检查首屏层级、字体、颜色、边框、圆角、选中态、状态图标和空白密度。
- 不允许恢复默认 shadcn 卡片网格、过多 badge、胶囊控件或大面积语义背景。
- Desktop 与 Mobile 均无裁切、溢出、不可读文本或意外换行。

## 不做

- 不重新设计或迁移 Compare、Review。
- 不引入深色模式。
- 不新增营销首页、插画、动态图表或装饰动画。
- 不改变推荐算法、评分规则或 golden query 内容本身。
- 不引入新的 Provider 配置能力。
- 不在页面内直接创建 GitHub Issue 或保存反馈自由文本。
- 不根据一次投票实时改分。
