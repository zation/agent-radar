# v0.7 P1 Golden Query Token Usage Implementation Plan

- Status: Approved
- Implementation commits: None
- Current status source: [docs/15-roadmap.md](../../15-roadmap.md)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Capture provider-reported token usage for every Golden Query request attempt, preserve it as reviewed release evidence, and show actionable totals in the GitHub Actions summary without changing recommendation behavior or public contracts.

**Architecture:** The recommendation provider adapter normalizes response-level usage and reports it through an optional internal observer. The Eval Runner owns case and retry identity, converts each actual provider call into one attempt record, and sends those records to a concurrency-safe collector. The build pipeline creates `reports/eval_token_usage.json`; preview finalization validates it, includes its checksum and compact summary in the artifact manifest, and renders the GitHub Summary from that validated manifest data.

**Tech Stack:** TypeScript, Node `node:test`, OpenAI-compatible chat-completions responses, existing artifact pipeline, preview manifest/checksum system, GitHub Actions Job Summary.

## Global Constraints

- Implement the approved design in [the v0.7 P1 Spec](../specs/2026-07-14-v0.7-p1-golden-query-token-usage-design.md); do not add token budgets, cost gates, prompt caching, batching, Top-K retrieval, independent Data Build, or Tool Card expansion.
- Keep `RecommendationResult`, HTTP, MCP, Web, and Eval Summary schemas unchanged. Usage is internal release evidence only.
- Count one attempt only after an actual provider request is initiated. A blocked-no-key suite has zero attempts.
- Preserve every retry. Sort artifact cases by `case_id` and attempts by one-based `attempt`, regardless of Eval Runner concurrency.
- Treat missing or malformed provider usage as non-blocking `unavailable`; treat usage artifact identity, schema, ordering, arithmetic, or checksum corruption as blocking.
- Never persist or display prompts, response bodies, raw provider errors, credentials, authorization headers, endpoints containing secrets, or inferred token counts.
- `npm run eval` remains a console-only Eval Summary command. `npm run pipeline` / Release Build owns the immutable usage artifact.
- Use `generatedAt` and `release` already supplied by the build; do not call `Date.now()` to create evidence identity.
- Before each commit run the focused tests and `git diff --check`. Preserve unrelated worktree changes.

```ts
export type ProviderUsageUnavailableReason =
  | "missing_provider_usage"
  | "invalid_provider_usage"
  | "request_failed";

export interface NormalizedProviderUsage {
  status: "reported" | "unavailable";
  input_tokens: number | null;
  cached_input_tokens: number | null;
  output_tokens: number | null;
  total_tokens: number | null;

export interface ProviderUsageObservation {

  usage: NormalizedProviderUsage;

export function normalizeProviderUsage(value: unknown): NormalizedProviderUsage;
export function unavailableProviderUsage(
  reason: ProviderUsageUnavailableReason,
): NormalizedProviderUsage;
```

Use own-property reads on an `unknown` object. Input, output, and total must all be finite, non-negative integers for `reported`. Cached input may be absent and then becomes `null`, but if present it must satisfy the same numeric validation. Do not check `total = input + output` and do not estimate missing fields.

- [ ] **Step 2: Define the artifact types**

Use these public exports from the new module:

```ts
export type EvalAttemptOutcome = "completed" | "provider_error" | "schema_error";
export type EvalAttemptFailureCategory = "none" | "provider_error" | "schema_error";
export type EvalCaseExecutionStatus = "completed" | "failed" | "blocked_no_key";

export interface EvalTokenUsageAttemptInput {
  case_id: string;
  attempt: number;
  provider: string;
  model_identifier: string;
  outcome: EvalAttemptOutcome;
  failure_category: EvalAttemptFailureCategory;
  usage: NormalizedProviderUsage;
}

export interface EvalTokenUsageCaseResult {
  case_id: string;
  execution_status: EvalCaseExecutionStatus;
}

export interface EvalTokenUsageArtifact {
  schema_version: "eval_token_usage.v1";
  generated_at: string;
  release: { release_id: string; commit_sha: string };
  summary: EvalTokenUsageSummary;
  cases: EvalTokenUsageCase[];
}

export class EvalTokenUsageCollector {
  constructor(input: {
    caseIds: string[];
    generatedAt: string;
    release: { release_id: string; commit_sha: string };
  });
  record(input: EvalTokenUsageAttemptInput): void;
  build(results: EvalTokenUsageCaseResult[]): EvalTokenUsageArtifact;
}

export function validateEvalTokenUsageArtifact(
  value: unknown,
  expectedRelease?: { release_id: string; commit_sha: string },
): EvalTokenUsageArtifact;
```

`EvalTokenUsageSummary` and `EvalTokenUsageCase` must contain every field approved in the Spec. The validator must parse from `unknown`, rebuild all case and suite aggregates from attempts, require exact case/attempt ordering, require unique case and attempt identities, and compare the optional expected release. It returns a fresh validated value, not the original mutable object.

- [ ] **Step 3: Implement concurrency-safe collection**

`record` performs a synchronous immutable copy into a private array, so concurrent async workers cannot partially mutate a record. `build` requires exactly one result state for every constructor case ID, groups and sorts a copied attempt array, derives cases, and calls the same validator used by finalization before returning.

Provider/model pairs may differ between attempts and are preserved per attempt. Token totals use only `usage.status === "reported"`; unavailable numeric fields must remain `null`.
