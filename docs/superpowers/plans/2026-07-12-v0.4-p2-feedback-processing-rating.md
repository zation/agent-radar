**Goal:** 将生产 D1 当前投票和经独立 LLM 分类的 GitHub Tool Feedback Issue 接入可回放的 `rating_result.v2`，并在 production approval 后、Worker 部署前幂等回写 Issue 状态。

**Architecture:** Build job 先从生产 D1 读取仅含 Tool 聚合值的快照，并以只读 GitHub 权限读取 open 待处理 Issue 与 closed accepted 历史；确定性代码完成解析、状态校验和去重，每个新 Issue 由一个隔离 LLM 请求分类，然后生成四类不可变 feedback artifacts 并参与评分。Deploy job 只消费已审核 processing plan，复查 Issue 并幂等 comment/label/close，全部成功后才部署同一 reviewed bundle。

**Tech Stack:** TypeScript 5、Node test runner、GitHub REST API、Cloudflare Wrangler/D1、现有 provider registry、GitHub Actions、React/Vite runtime artifacts。

## Global Constraints

- 固定仓库为 `zation/agent-radar`，不得读取或写入其他仓库。
- 规则版本固定为 `feedback_rules.v0.1`，分类器版本固定为 `feedback_classifier.v0.1`。
- 每个未处理 Issue 一个 LLM 请求，最大并发 `4`，每次 release 最多 `50` 个新 Issue，单项失败最多重试 `1` 次。
- D1 投票与 accepted Issue 独立相加：D1 每票 `±0.2`，Issue 每条 `±1`；Tool adjustment clamp 为 `[-3, 3]`，final score clamp 为 `[0, 100]`。
- Issue 流按 GitHub numeric user ID + Tool ID 去重，以 `created_at`、Issue number 决定最新项。
- feedback 不得修改 dimension scores、risk、trust、evidence 或 critical safety gate。
- Bundle 不保存 D1 用户级行、完整 prompt、provider 原始响应、OAuth token、Cookie、secret 或 Issue 原始自由文本。
- Build job 不写 GitHub；Deploy job 必须在 production approval 后回写，并在 Worker deploy 前完成。
- 不增加付费服务或运行时依赖；沿用现有 provider registry、Wrangler 和 Node 标准库。

- Spec coverage: Tasks 1–5 cover input selection, deterministic validation, LLM isolation, state machine, dedupe and privacy; Tasks 6–8 cover score/rating/artifacts/evidence; Tasks 9–10 cover approval-gated idempotent writeback; Task 11 covers authoritative docs and production acceptance.
