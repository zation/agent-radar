- Status: Completed

- Current status source: [docs/15-roadmap.md](../v0.x-roadmap.md)

## 背景

Agent Radar 已在同一个 Cloudflare Worker 中提供 `/api/mcp`，支持 `initialize`、`tools/list` 和四个只读 `tools/call`。当前实现由 `src/api/handler.ts` 手写 JSON-RPC 协议分发，能够通过现有 smoke，但协议版本、参数校验、工具注册和错误语义需要由项目自行维护；它也尚未发布到官方 MCP Registry。

v0.6 把远程 MCP 协议层迁移到官方 TypeScript SDK v2，并把已经验证的生产 endpoint 作为 remote-only server 实际发布到官方 MCP Registry。此次变更不把 Agent Radar 改造成安装市场，不新增第三方工具执行能力，也不发布没有运行价值的 npm/stdio wrapper。

发布修订（2026-07-14）：不可变 tag `all-v0.6.0`、`all-v0.6.1`、`all-v0.6.2` 分别因 provider HTTP 402、MiniMax 中国区 base URL 未注入、以及默认 endpoint 单元测试未隔离 CI override 而在生产部署前失败，Registry 均未发布。依照不可移动 tag 和补丁版本规则，实际发布目标推进为 `all-v0.6.3` / Registry `0.6.3`；产品范围和架构不变。

截至 2026-07-13，官方 v2 package 已发布 `2.0.0-beta.3`，API 可用于 Web Standard runtime，但 stable v2 仍预计随 2026-07-28 MCP specification 在 2026 Q3 发布。项目接受 beta 风险，精确锁定版本并通过协议、Worker 和生产门禁控制；stable v2 升级是后续独立变更。

## 目标

1. 使用官方 `@modelcontextprotocol/server@2.0.0-beta.3` 实现 `/api/mcp` 的无状态 Streamable HTTP。
2. 保持 `search_tools`、`get_tool_card`、`recommend_tools` 和 `explain_rating` 四个业务能力，同时消除 HTTP 与 MCP 业务逻辑分叉。
3. 删除请求 body 和 MCP tool arguments 中的 LLM `api_key`，改用 request-scoped secret header。
4. 创建并验证 remote-only `server.json`，以 `io.github.zation/agent-radar` 发布到官方 MCP Registry。
5. 只在同一 tag/SHA 的生产 Worker、reviewed bundle、评测和 MCP smoke 全部成功后，通过 GitHub OIDC 发布 Registry metadata。
6. 生成可追溯的 Registry publication evidence，并在官方 Registry API 中验证实际可发现性。

## 非目标

- 不实现 MCP OAuth、Agent Radar 账号或企业 SSO。
- 不实现 stateful MCP session、resumability、server notification、resources 或 prompts。
- 不发布 npm MCP package、stdio server、Docker image 或本地安装 wrapper。
- 不增加自动安装、授权或执行第三方工具的能力。
- 不重做 Web UI 视觉，也不引入完整 Provider 配置页面。
- 不保留 body/tool argument `api_key` 的兼容路径。
- 不把官方 MCP Registry 接入 Agent Radar ingestion Source Registry；v0.6 只发布自身 server。
- 不在 v0.6 内追踪 beta 后的任意 SDK 变化；升级必须是显式、受测的独立变更。

## 已选方案

采用 remote-only Registry publication：生产 Worker 是唯一运行实体，`server.json` 只声明 Streamable HTTP remote。该方案比新增 npm wrapper 更符合当前部署事实，也比手工发布提供更强的版本、SHA 和证据绑定。

未选方案：

- npm/stdio wrapper：会增加第二套运行入口、包发布凭证和兼容测试，没有用户价值。
- 手工 Registry 发布：首次操作简单，但无法可靠绑定 reviewed production evidence，后续也难以重放和审计。
- 把 Registry publish 直接放进 production deploy job：若 Registry 失败，已经成功的生产部署会被同一 job 标为失败，混淆两种恢复语义。

## 组件边界

### Tool Contract

单一 Tool Contract 定义四个工具的名称、描述、输入 schema、输出 schema、只读属性和结果包装要求。以下消费者从同一 contract 派生：

- SDK v2 的 MCP tool registration。
- `/api/mcp_manifest`。
- pipeline 生成的 `mcp_tools.json`、examples 和 smoke checklist。
- 单元测试与部署后 smoke 的 expected tool set。

Tool Contract 不访问 repository、provider 或 Worker environment。

### Tool Service

Tool Service 承载当前位于 HTTP handler 中的搜索、Tool Card 查询、推荐和评分解释业务。它接收已验证的领域输入和显式 request context，返回现有稳定业务对象。

HTTP API adapter 与 MCP adapter 都调用 Tool Service；二者不得复制查询、评分或推荐逻辑。Tool Service 依赖 `ToolRepository`、recommendation engine 和 provider registry，不依赖 MCP transport。

### HTTP API Adapter

现有 `/api/search_tools`、`/api/get_tool_card`、`/api/recommend_tools` 和 `/api/explain_rating` 保留。Adapter 负责读取 query/body/header、构造 request context、调用 Tool Service，并映射 HTTP 状态与 JSON error。

Web Recommend UI 继续在内存中接收用户的 Provider key，但请求时改为 `X-Agent-Radar-LLM-API-Key` header；key 不进入 body、URL、本地持久化或返回 payload。

### MCP Adapter

`/api/mcp` 使用 `@modelcontextprotocol/server@2.0.0-beta.3` 的 `createMcpHandler` Web Standard、无状态 Streamable HTTP handler。SDK 负责 protocol negotiation、JSON-RPC envelope、tool registration、schema validation 和标准协议错误。Handler 保留 SDK 默认的 2025-era stateless compatibility；v0.6 的 contract 不依赖 2026-07-28 draft-only method 或 capability，draft 最终确定后的显式采用另行评审。

MCP Adapter 负责：

- 从 Tool Contract 注册四个只读 tools。
- 从 request header 建立 credential context。
- 调用 Tool Service。
- 同时返回 JSON text content 与 `structuredContent`，让简单客户端和结构化客户端消费同一业务结果。
- 把业务错误映射为 `isError: true` 的安全 Tool Result。

不启用 session ID、SSE notification、GET stream 或 DELETE session。非 POST MCP 请求返回明确 method error；现有 OPTIONS 仅用于 CORS preflight。

### Registry Metadata

仓库根目录维护 `server.json`：

- `name`: `io.github.zation/agent-radar`
- `title`: `Agent Radar`
- `version`: `0.6.3`
- `repository.url`: `https://github.com/zation/agent-radar`
- `repository.source`: `github`
- `remotes[0].type`: `streamable-http`
- `remotes[0].url`: `https://agent-radar.zation1.workers.dev/api/mcp`
- `remotes[0].headers[0]`: `{ "name": "X-Agent-Radar-LLM-API-Key", "description": "LLM provider API key used only by recommend_tools.", "isRequired": false, "isSecret": true, "format": "string" }`
- 不包含 `packages`

`server.json` 使用官方当前 Registry schema `https://static.modelcontextprotocol.io/schemas/2025-12-11/server.schema.json`，并由测试固定 schema URL、name、version、remote URL、transport、repository、header secret 属性和无 packages 约束。Registry schema 变化必须通过后续显式变更升级，不能在 workflow 中动态替换。Registry metadata 不宣称 Agent Radar 能安装或运行所推荐的工具。

## Credential 设计

LLM credential 统一从 `X-Agent-Radar-LLM-API-Key` 读取。`recommend_tools` 的普通 `model` 仍是 tool/body 输入，provider registry 先根据 model 解析 Provider，再把 header key 交给 provider client。

解析优先级：

1. 当前请求的 `X-Agent-Radar-LLM-API-Key`。
2. 部署环境显式配置的 `AGENT_RADAR_LLM_API_KEY` fallback。
3. 返回稳定 `missing_provider_key` 错误。

v0.6 不新建或自动配置生产 Worker fallback secret；若维护者未来配置它，必须另行评估公开调用成本和滥用边界。

请求 body 或 MCP arguments 出现 `api_key` 时必须拒绝，并说明使用 secret header；不得静默忽略或继续读取。密钥不得进入日志、telemetry、error body、MCP content、artifact、D1、cache key 或 production evidence。

Registry remote header 声明为 optional + secret，因此只使用搜索、查询和评分解释的客户端无需提供 Provider key。

## 请求安全边界

- `/api/mcp` 请求 body 上限为 65,536 UTF-8 bytes；超过上限返回 HTTP 413，不进入 SDK 或 Tool Service。
- 生产 Host 必须匹配请求 URL 的固定 Worker host；本地开发只允许显式配置的 localhost host。
- 无 `Origin` 的原生 MCP client 正常通过。
- 存在 `Origin` 时只允许同源或显式 allowlist；非法或不可解析 Origin 返回 403。
- CORS 允许 `content-type` 与 `x-agent-radar-llm-api-key`，不允许 credential echo。
- DELETE、PUT、PATCH 和未知 write-like tool 均被拒绝。
- Provider upstream error 只保留稳定 code、provider 和必要 status；不返回上游原始正文或 stack。

## 数据流

```text
MCP client
  -> POST /api/mcp
  -> host/origin/size guard
  -> SDK v2 stateless Streamable HTTP
  -> Tool Contract schema validation
  -> Tool Service
  -> repository or recommendation engine
  -> structuredContent + JSON text content
```

```text
Web/HTTP client
  -> /api/<tool> + optional LLM key header
  -> HTTP adapter
  -> same Tool Service
  -> existing stable business JSON
```

```text
all-v0.6.3 tag
  -> Release All reviewed bundle
  -> production approval
  -> deploy Worker
  -> MCP smoke + production_release_evidence.json
  -> separate successful workflow_run
  -> validate tag/SHA/server.json/online endpoint
  -> GitHub OIDC
  -> mcp-publisher publish
  -> Registry API verification
  -> mcp-registry-publication-evidence.json
```

## 版本绑定

Registry 使用 SemVer，不使用 `all-v` 前缀：

- `all-v0.6.3` 对应 `server.json` `0.6.3`。
- `all-v0.6.3` 对应 `0.6.3`。

发布 workflow 必须验证 `server.json.version === release_tag` 去掉 `all-v` 后的值。Registry version 一旦发布不可修改；同一版本若已存在，只有 Registry 查询返回完全一致的 name、version、remote URL 和 repository 时才可作为幂等成功。任何差异都阻断并要求新的 patch release。

SDK 的 serverInfo version 使用同一 SemVer；生产 `/api/version.release_id` 继续使用完整 `all-v0.6.3`。

## Registry 发布自动化

新增独立 workflow，自动监听 `Release All` 的 successful completion，并支持相同 run ID 的手动重试。它必须：

1. 拒绝非本仓库、非 `all-v*` tag、非 success conclusion 或缺少 production evidence 的 run。
2. checkout evidence 中的精确 SHA，不使用浮动 `main`。
3. 下载 `production-release-evidence.json` 和 MCP smoke artifact。
4. 验证 run ID、tag、SHA、Worker URL、MCP endpoint、smoke 4/4 和 reviewed bundle checksum 互相一致。
5. 验证线上 `/api/version` 与 `server.json` 版本映射，并重新运行 MCP smoke。
6. 使用 `permissions: { contents: read, actions: read, id-token: write }`，通过 `mcp-publisher login github-oidc` 获取短期身份；不保存 MCP PAT。
7. 使用固定的 `mcp-publisher` `v1.8.0` Linux amd64 artifact，并验证提交在 workflow 中的 SHA-256 `1370446bbe74d562608e8005a6ccce02d146a661fbd78674e11cc70b9618d6cf`；不得下载 `latest`。
8. 发布后查询 `https://registry.modelcontextprotocol.io/v0.1/servers?search=io.github.zation/agent-radar`，精确选择 name/version，验证 remote URL 与 transport。
9. 生成并上传 publication evidence，包含 source run、SHA、tag、Registry name/version、published metadata checksum、remote endpoint、query timestamp 和验证结果。

Registry publish failure 不回滚 Worker，也不篡改成功的 production evidence；Roadmap 保持 v0.6 未完成，workflow 可在不修改 metadata 的前提下重试。

## 错误处理

- 协议 envelope、method、版本和 schema 错误由 SDK 返回标准 MCP/JSON-RPC error。
- Tool Service 的已知业务错误映射为稳定 code、可读 message 和 recovery advice，并通过 MCP `isError: true` 返回。
- 未知 Tool ID、Rating 缺失和无匹配结果保持现有保守语义，不转换为成功空对象。
- Provider 401/403、429、model unavailable、timeout 和 schema failure 保持现有分类，但必须经过 credential redaction。
- Registry schema、OIDC、namespace、publication 或 query verification 任一失败均使 Registry workflow 失败。
- Registry 已存在同版本但 metadata 不同视为不可恢复冲突，必须发布新 patch version。

## 测试设计

### 单元与 contract

- 四个 Tool Contract 名称、description、input schema 和 read-only metadata。
- Tool Service 对搜索、Tool Card、推荐和评分解释的成功与失败路径。
- HTTP adapter 与 MCP adapter 对相同业务输入返回相同核心对象。
- MCP `structuredContent` 与 JSON text content 表达同一对象。
- body/tool argument `api_key` 被拒绝；header key 正常传递；所有输出和捕获日志不含测试 secret。
- manifest、pipeline artifacts、SDK registration 和 smoke expected names 从同一 contract 派生。

### 协议与 Worker

- `initialize`、`tools/list`、四个 `tools/call` registration、invalid params、unknown method 和 unknown tool。
- 2025-era stateless client compatibility；测试不以 2026-07-28 draft-only capability 作为 v0.6 验收条件。
- stateless Streamable HTTP 的 POST/OPTIONS 行为，以及 GET/DELETE/PUT/PATCH rejection。
- Host、Origin、CORS 和 65,536-byte request boundary。
- Wrangler dry-run 证明 SDK v2 bundle 可运行于 Cloudflare Worker，不依赖 Node-only transport。
- Worker bundle 不包含未使用的 client、Express、stdio 或 Node transport adapter。

### 发布与线上

- `npm test`、ESLint、Stylelint、TypeScript build、Vite build 和 Wrangler dry-run。
- Release All 的真实 provider golden eval 24/24、critical safety 4/4、data quality、promotion 和 review checksum gates。
- 部署后 smoke 覆盖 initialize、tools/list、搜索/查询/评分调用、推荐缺 key 的安全错误和 write rejection。
- Registry workflow fixture 覆盖正确 evidence、错误 tag/SHA、失败 release、server version mismatch、已存在一致版本和已存在冲突版本。
- 官方 Registry API 返回 `io.github.zation/agent-radar`、`0.6.3`、`streamable-http` 和生产 `/api/mcp`。

## 文档与完成状态

实现完成时必须在同一变更更新 `README.md`、需求、架构、安全、部署、评测和 Roadmap 中的当前事实。不得在部署或 Registry 查询成功前把 v0.6 写为 Completed。

v0.6 完成必须同时具备：

- 代码和 contract tests 通过。
- `all-v0.6.3` production release 成功。
- 线上 `/api/version`、MCP SDK endpoint 和 eval metadata 一致。
- MCP smoke 通过。
- 官方 Registry publication 成功且 API 查询验证通过。
- production evidence 与 Registry publication evidence 可下载、checksum 可验证。

## 回滚

Registry publication 是最后一步。若 publication 前 Worker 失败，保持 `all-v0.5.1` 为最后完整基线，不发布对应 Registry 版本。

若 Registry 版本发布后仍必须回滚 Worker，Registry metadata 不可覆盖。运维记录与 Roadmap 标记对应 endpoint 已回滚，修复后递增 patch 版本恢复一致性。

## 验收标准

- `/api/mcp` 由精确锁定的官方 SDK v2 Web Standard handler 提供，不再手写 JSON-RPC dispatcher。
- 四个工具通过单一 contract 和 Tool Service 同时服务 HTTP 与 MCP。
- LLM key 只从 secret header 或显式 environment fallback 读取，旧 `api_key` 输入被拒绝且无泄漏。
- remote-only `server.json` 通过官方 schema 和项目 contract 验证。
- `all-v0.6.3` 生产发布、24/24 eval、4/4 critical safety 和部署后 MCP smoke 通过。
- GitHub OIDC Registry workflow 使用固定 publisher binary，发布并验证 `io.github.zation/agent-radar@0.6.3`。
- Roadmap、领域权威文档和两类 release evidence 与实际线上状态一致。

## 风险与缓解

- v2 beta API 变化：精确锁定 `2.0.0-beta.3`，不使用 floating beta；stable 升级单独评审。
- SDK 引入 Node-only 依赖：只导入 Web Standard server entry，并以 Wrangler dry-run 和 bundle inspection 阻断。
- Header key 在浏览器内存可见：不持久化、不进 body/URL/log；浏览器端 secret 无法对用户自身脚本隐藏。
- 可选 Worker fallback 造成公共调用成本：v0.6 不自动配置 fallback；未来配置需独立成本和滥用评估。
- Registry preview 可能重置或变更：保存 publication evidence，并以官方 API 实时查询作为当前状态事实。
- OIDC workflow 供应链风险：最小权限、精确 SHA checkout、固定 publisher version 和已提交 checksum。
- Registry immutable metadata 与生产回滚不一致：publish last；发生回滚时用新 patch version 修复，不覆盖历史。
