## 目标

在不改变 `tool_card.v1` 核心字段语义和 reviewed bundle 发布模型的前提下，交付字段 provenance v2、跨来源冲突报告、URL checker v2、数据质量报告和 Review Summary v2，并把可靠发布数据从 29 张扩展到 50–150 张 Tool Cards。

## 实施原则

- `docs/15-roadmap.md` 仍是阶段状态唯一事实源；本文件只记录 P1 的执行步骤和验证证据。
- `tool_card_field_value_provenance.v1` 与 `tool_card_url_validation.v1` 保留一个迁移周期；新增 v2 文件，不就地改变旧 artifact 语义。
- 所有事实必须来自 Source Record、Override Record 或确定性派生；LLM 不能成为字段事实来源，也不能解除 gate。
- 新增来源使用精确官方仓库或官方 package metadata；热度只用于发现，不能提高事实可信度。
- P1 的阻断判断由确定性代码完成。关键字段 provenance 不完整、关键语义冲突、关键 URL 永久失败或摘要 checksum 不一致时必须失败。
- 每个新 artifact 都要有 schema/version、生成时间、规则版本、summary、稳定 reason code 和对应测试。

## 目标数据流

```text
Source Registry + Raw Snapshots
  -> Source Records
  -> normalizeToolCardDraftsWithEvidence
       -> Tool Card drafts
       -> field candidates / selections
       -> conflict decisions
  -> provenance v1 + v2
  -> conflict report v1
  -> dedup + validation + URL validation v1 + v2
  -> auto review + release admission + promotion check
  -> data quality report v1
  -> review summary v2
  -> reviewed bundle + manifest checksum verification
```

### 3.1 先写失败测试

使用注入的 fake fetch，不访问公网。覆盖：

- 2xx/3xx 为 `reachable`，记录 `final_url` 和 redirect chain。
- HEAD 返回 405/501 时只回退一次最小 GET。
- 401/403 为 `auth_required`；429 为 `rate_limited`；404/410 为 `permanent_failure`；5xx、超时、DNS/TLS 异常为 `transient_error`。
- 仅 429、5xx 和瞬时网络错误重试，默认最多 2 次，固定短退避可由测试注入为零。
- URL 含凭证、跳转到非 HTTP(S) 或明显不同的登录/广告域名时，不继续请求并输出稳定 reason code。
- 读取上一版 artifact 后更新 `consecutive_failure_count` 和 `last_success_at`；没有历史时 `history_status` 为 `no_baseline`。
- URL 检查未启用时输出 `skipped`，但不能伪装为成功。

### 3.2 实现 checker

定义：

```ts
export type ToolCardUrlValidationStatusV2 =
  | "reachable"
  | "permanent_failure"
  | "auth_required"
  | "rate_limited"
  | "transient_error"
  | "skipped";

export interface ToolCardUrlValidationItemV2 {
  tool_id: string;
  field_path: string;
  url: string;
  status: ToolCardUrlValidationStatusV2;
  reason_code: string;
  method: "HEAD" | "GET" | "none";
  http_status?: number;
  final_url?: string;
  redirects: string[];
  checked_at: string;
  attempt_count: number;
  consecutive_failure_count: number;
  last_success_at?: string;
  history_status: "continued" | "no_baseline";
  critical: boolean;
}

export interface ToolCardUrlValidationArtifactV2 {
  schema_version: "tool_card_url_validation.v2";
  generated_at: string;
  options: { enabled: boolean; timeout_ms: number; max_retries: number };
  items: ToolCardUrlValidationItemV2[];
  summary: Record<ToolCardUrlValidationStatusV2, number> & {
    blocking: number;
    stale: number;
  };
}
```

默认 `timeout_ms = 5000`、`max_retries = 2`。检查范围包括 `source_urls`、`install_methods[*].docs_url`，以及 schema 中实际存在的 repo/homepage/package/docs URL。对 URL 去重请求，但每个 field path 单独保留结果。

迁移期继续生成 v1 artifact；v2 使用新文件 `public/data/tool_card_url_validation.v2.json`。上一版本读取顺序为显式 CLI option、已恢复 reviewed artifact、无基线，禁止静默读取任意工作区旧文件。

promotion 阻断规则：关键 URL 的 404/410、DNS/TLS 永久错误、带凭证 URL或异常 redirect 立即阻断；其他瞬时错误先 warning，连续两次发布仍失败时转为 blocking。401/403 和 429 保持可审核 warning，除非该卡不存在其他公开证据 URL。

## 已知风险与控制

- **GitHub rate limit**：429 只进入可解释 warning，并保留连续状态；CI 使用最小权限 token 时不得写入 artifact。
- **URL 波动**：瞬时失败不立即永久否定工具，但连续失败会升级；历史基线缺失必须显式报告。
- **来源扩容带来噪声**：只启用精确官方来源，所有新增卡片走同一 admission 和 promotion gate。
- **artifact 版本分叉**：v1/v2 并行输出且都进 manifest，消费者迁移完成前不删除 v1。
- **Review Summary 自引用 checksum**：采用两阶段 manifest，摘要只引用其输入 artifact，最终 manifest 再记录摘要 checksum。
