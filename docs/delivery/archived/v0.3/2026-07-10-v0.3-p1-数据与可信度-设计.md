## 目标

把 v0.2 的最小可靠发布链路提升为可回放、可解释、能阻断关键数据质量问题的 v0.3 数据可信度基线，同时把可靠发布数据扩展到 50–150 张高质量 Tool Cards。

P1 回答五个问题：

1. 每个关键字段来自哪些来源和值？
2. 多来源冲突时为何选择当前值？
3. 作为证据和安装入口的 URL 当前是什么状态？
4. 发布审核者能否只看一份摘要定位所有阻断项？
5. 扩大 Tool Card 覆盖后，数据质量是否仍满足发布门槛？

## 当前基线

v0.2 已实现：

- Source Registry 驱动的 crawl、parse、normalize、dedup、auto review、release admission 和 promotion check。
- `tool_card_field_provenance.v1`，覆盖 `permissions`、`security` 和 `maintenance` 的 schema 级证据信号。
- `tool_card_field_value_provenance.v1`，记录单个主要 Source Record 字段路径、原始值预览、归一化值预览和 Override Record。
- `tool_card_url_validation.v1`，支持可选 HEAD/GET 检查；默认未启用外网检查。
- intervention requests、Source Registry review requests、review markdown、artifact manifest 和 GitHub Actions compact summary。
- 29 张已验证 Tool Cards。

当前不足：

- 字段值 provenance 默认依赖第一个 `evidence_ref`，不能完整表达多来源合并。
- 没有统一记录转换规则版本、候选值、冲突类型和最终选择依据。
- URL 检查缺少超时、有限重试、细分状态、检查新鲜度和连续失败处理。
- 审核证据分散在多个 artifact，reviewer 仍需人工拼接阻断原因。
- 数据扩张时缺少统一的数据质量报告和跨版本趋势摘要。

## 范围

### 1. Field provenance v2

新增 `tool_card_field_value_provenance.v2`，每个最终字段至少记录：

- `tool_id`
- `tool_card_field`
- 一个或多个 `source_record_id`
- 每个来源的 `source_id`、`source_field_path`、原始值摘要或 hash
- `normalized_value_preview`
- `transformation_type`：`copy`、`normalize`、`merge`、`derive` 或 `override`
- `parser_version` 和 `normalizer_version`
- `source_confidence`、来源更新时间和抓取时间
- `selected` 状态及选择理由
- 相关 `override_record_id` 和 evidence refs

数组、嵌套字段和推导字段不能只记录顶层字段名；必须能定位参与生成最终值的输入项。

关键字段至少包括：

- `type`
- `summary`
- `source_urls`
- `install_methods`
- `permissions`
- `security`
- `maintenance`
- `license`
- `confidence`

已发布候选的关键字段 provenance 覆盖率必须为 100%。

### 2. 跨来源冲突与实体合并

新增 `tool_card_conflict_report.v1`，记录：

- canonical repo、package 和 docs identity。
- alias/package key 映射。
- 同一字段的候选值与来源。
- 冲突类型：格式差异、时效差异、可信度差异、语义冲突或 override。
- 自动选择规则、最终选择和理由。
- 无法自动解决时的 intervention request。

默认选择顺序：

1. 有公开证据的 Override Record，仅覆盖其明确声明的字段。
2. 与字段语义直接对应的官方来源。
3. 精确 repository/package metadata。
4. 受控发现来源。
5. 更高 source confidence。
6. 在同等可信度下选择更新且非空的证据。

来源数量、stars 或 package 存在不能单独覆盖官方安全、权限或维护事实。关键字段语义冲突无法解释时必须阻止 promotion；非关键展示字段冲突可以 warning，但必须出现在 Review Summary。

### 3. URL checker v2

新增 `tool_card_url_validation.v2`，覆盖：

- `source_urls`
- `docs_url`
- `repo_url`
- `homepage_url`
- `package_urls`
- `install_methods.docs_url`

状态分类：

- `reachable`
- `permanent_failure`
- `auth_required`
- `rate_limited`
- `transient_error`
- `skipped`

检查策略：

- 每次请求有明确超时。
- HEAD 不受支持或结果不足时回退到最小 GET。
- 仅对瞬时网络错误、429 和 5xx 做有限重试。
- 记录 HTTP 状态、最终 URL、重定向、检查时间、失败原因和检查方法。
- 如果能读取上一版 URL validation artifact，记录连续失败次数和最后成功时间；缺少历史 artifact 时明确标记无趋势基线。
- 不跟随到非 HTTP(S)、带凭证或明显无关的异常目标。

关键 URL 的持续 404/410、证书/域名错误或异常跨域重定向必须阻止 promotion。401/403、429 和瞬时错误不得直接等同永久失效，但必须有可解释状态并进入审核摘要。

### 4. Review Summary v2

新增统一的 `review_summary.v2`，聚合：

- Source Registry added、removed、changed 和 review requests。
- Tool Card added、changed、removed。
- 关键字段 provenance coverage。
- 字段冲突、选择结果和未解决项。
- URL 状态和连续失败。
- duplicate signals、intervention requests 和 Override Records。
- validation errors/warnings。
- auto review、release admission、promotion candidates 和 promotion check。
- artifact checksums、数据版本、规则版本和 git SHA。

Review Summary 默认只突出发生变化或需要行动的项目，同时保留到完整 artifact 的路径。任何阻断项必须具有稳定 reason code、对象 ID、证据路径和 suggested action。

### 5. 数据质量报告与覆盖扩展

新增 `data_quality_report.v1`，至少统计：

- Tool Card 总数和类型分布。
- 必填字段完整率。
- 关键字段 provenance 覆盖率。
- source confidence 分布。
- unknown permission、security 和 maintenance 比例。
- 重复候选率和未解决冲突数。
- URL 状态分布和检查新鲜度。
- parser warnings、intervention requests 和 promotion 阻断数。
- 与上一发布版本的变化。

扩展目标为 50–150 张高质量 Tool Cards。新增来源必须优先覆盖当前 golden query 和高价值开发场景；不得为了数量降低来源、provenance、URL 或 promotion 门槛。

## 数据流

```text
enabled Source Registry
  -> crawl + immutable Raw Snapshot
  -> parse to Source Records
  -> canonical identity + alias/package keys
  -> normalize field candidates
  -> conflict resolution + provenance v2
  -> dedup + validation
  -> URL checker v2
  -> auto review + intervention requests
  -> release admission + promotion check
  -> data quality report
  -> Review Summary v2
  -> immutable reviewed bundle
```

P1 不改变 build-once、review-once、原样部署 reviewed bundle 的发布模型。

## 发布门禁

以下情况阻止可靠发布：

- 关键字段 provenance coverage 低于 100%。
- 存在未解决的关键字段语义冲突。
- 存在未解决 published/incoming duplicate。
- 关键 URL 出现未解释的永久失败或异常重定向。
- 存在 validation error、blocked release admission、blocked promotion candidate 或待处理 intervention request。
- Review Summary、manifest 与实际 artifact checksum 不一致。

以下情况默认 warning：

- 非关键展示字段冲突。
- 第一次出现的瞬时 URL 错误。
- 需要登录的官方文档 URL，但存在其他公开证据。
- 缺少上一版本趋势基线。

## 兼容与迁移

- P1 优先新增派生 artifact，不改变 `tool_card.v1` 核心字段语义。
- v2 artifact 与 v1 并行输出一个迁移周期，所有消费者和测试切换后再停止生成 v1。
- 如果实现需要修改 Source Record、Override Record 或 Tool Card 核心 schema，必须先更新 `docs/04-data-model.md` 并单独确认语义变化。
- 现有 API/MCP/Web 继续读取可靠 Tool Card 和 Rating artifacts，不直接消费未通过 gate 的候选。

## 安全边界

- 不自动信任新来源或仅凭热度提升 source confidence。
- 不抓取私有仓库、登录后页面、用户文件、邮件或浏览器数据。
- URL checker 不发送 Cookie、Authorization、API key 或其他用户凭证。
- Override Record 必须包含公开 evidence、修改理由、reviewer 和时间。
- LLM 可以生成审核摘要，但不能成为字段事实来源，也不能解除确定性 validation、conflict 或 promotion gate。

## 非目标

P1 不包含：

- 安全风险评分 v1、结构化 Human Approval 或 Eval Diff；这些属于 v0.3 P2。
- GitHub OAuth、D1 投票、Issue Form 或反馈评分；这些属于 v0.4。
- UI 视觉重构；它属于 v0.4 P1。
- Provider 配置 UI、浏览器运行时 Registry 或 direct/proxy 决策；这些属于 Backlog。
- 自动定时采集、自动信任来源或自动发布未知 Tool Card。

## 验证

P1 Plan 必须覆盖：

- provenance v2 的多来源、数组、嵌套、derive 和 override 测试。
- 冲突自动选择、无法解决阻断和 alias/package merge 测试。
- URL timeout、HEAD/GET fallback、redirect、429、5xx、401/403、404/410 和异常目标测试。
- Review Summary actionable-only 和 checksum 一致性测试。
- 数据质量指标和跨版本 diff 测试。
- promotion/release workflow contract tests。
- 完整 `npm test`、lint、stylelint、production build 和 reviewed bundle 验证。

## 完成标准

- 发布数据包含 50–150 张通过 promotion gate 的高质量 Tool Cards。
- 已发布候选关键字段 provenance coverage 为 100%。
- 未解决关键字段冲突、重复项、validation error、blocked promotion 和 intervention request 均为 0。
- 关键 URL 均有近期 reachable 或可解释状态，没有未解释的持续永久失败。
- `data_quality_report.v1` 和 `review_summary.v2` 能独立定位所有 warning、阻断项和证据 artifact。
- reviewed bundle 的数据、摘要、manifest、checksums 和最终部署输入一致。
