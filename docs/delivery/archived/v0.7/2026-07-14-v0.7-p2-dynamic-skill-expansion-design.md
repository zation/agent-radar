# v0.7 P2: Dynamic Skill Tool Card Expansion Design

- Status: Completed
- Implementation commits: `854b6fad`, `013e6ed4`, `3970686a`, `9eec81a0`, `b4e3b5dd`, `33efb4ce`, `e8ed8b87`, `d8b4aa61`, `16c654a2`, `0fbdcc6c`
- Current status source: [docs/15-roadmap.md](../v0.x-roadmap.md)

## Background

Agent Radar v0.7 P1 established real-provider token usage evidence before catalog growth. The current production baseline contains 53 Tool Cards and `all-v0.7.0` records 639,795 total tokens for 24 Golden Queries. Dynamic discovery currently exists only for the `github-topic-mcp` source, whose GitHub query is embedded in crawler code and whose search result repository is treated as one Tool Card candidate.

P2 expands Skill coverage without hard-coding individual Skill names. GitHub repositories tagged `agent-skills` commonly expose multiple independent Skills below a root `skills/` directory. The discovery unit is therefore a repository, while the Tool Card unit is one eligible `SKILL.md` file. The first controlled batch reads every eligible Skill from the two highest-starred repositories returned by `topic:agent-skills` at build time.

The current Auto Review and Rating Result envelopes are shared across Tool Card types. P2 preserves those common storage contracts. It adds an internal common finding interface for deterministic ingestion checks and a common rating-policy interface whose first type-specific implementation is Skill. It does not create a `skill_review`, `skill_quality_report`, or type-specific Tool Card schema.

## Goals

1. Add an enabled, configurable `github-topic-agent-skills` Source that discovers the two highest-starred public repositories tagged `agent-skills`.
2. Enumerate every eligible root `skills/**/SKILL.md` in those repositories and generate one Skill Tool Card draft per file.
3. Keep Skill identity stable by repository and Skill path, even when repository ranking or metadata changes.
4. Treat only the current build's successfully fetched Skill records as this Source's authoritative dataset, without previous-build fallback or accumulation.
5. Preserve `tool_card.v1`, `tool_card_auto_review.v1`, and `rating_result.v2` as the common persisted envelopes.
6. Route deterministic ingestion findings through one internal representation and map them to the existing Auto Review fields.
7. Introduce a shared rating-policy boundary and implement the documented Skill dimensions and weights without changing non-Skill rating behavior in this batch.
8. Run the complete provenance, conflict, duplicate, promotion, Golden Query, critical-safety, and token-usage evidence path against the expanded catalog.

## Non-Goals

- Adding exact repository sources for individual Skill collections.
- Querying `topic:skills`, GitHub Code Search, skills.sh, or another external Skill registry.
- Adding CLI or Prompt/Rules sources; those remain later P2 increments.
- Limiting this batch to four Tool Cards. The selected design reads all eligible Skills from the top two repositories.
- Trusting, installing, cloning, or executing repository content.
- Fetching private repositories, using cookies, or requiring a GitHub token.
- Creating a type-specific Auto Review artifact, quality report, Tool Card schema, or Rating Result schema.
- Migrating MCP, Agent, Framework, CLI, or Prompt/Rules cards to new rating dimensions in this batch.
- Independent Data Build, scheduled ingestion, automatic production deployment, prompt caching, batching, provider sessions, or Top-K retrieval.
- Adding a token-cost release threshold. P2 observes the delta from P1 but does not reject solely on cost.
- Merging previous-build Skill records, accumulating repositories that leave the top two, or retaining failed Skill records.

## Selected Source

The Source Registry adds:

```text
id: github-topic-agent-skills
url: https://github.com/topics/agent-skills
parser: github_skill_topic_parser
covered_tool_types: [skill]
collection_method: api
trust_level: active_open_source
recommended_frequency: weekly
failure_policy: keep successful records from the current crawl only
```

Its `github_discovery` configuration is data, not crawler constants:

```ts
github_discovery: {
  query: "topic:agent-skills",
  sort: "stars",
  order: "desc",
  repository_limit: 2,
  expansion: {
    kind: "skill_manifests",
    root: "skills/",
    manifest: "SKILL.md"
  }
}
```

The generic GitHub topic request builder also receives explicit query, sort, order, and result-limit configuration. `github-topic-mcp` is migrated to the same configuration while preserving its exact current request semantics: `topic:mcp`, stars descending, 20 results. A regression fixture must prove that migration does not change its request URL or results.

The top-two selection is recalculated for each build. At design time, the current repositories are `anthropics/skills` and `DietrichGebert/ponytail`, yielding 17 and 6 eligible root Skills respectively. Those names and the current total of 23 are evidence for expected first-batch size, not hard-coded configuration or a permanent contract.

## Repository and Skill Granularity

The repository search result is discovery metadata, not a Tool Card. For each selected repository, the crawler reads the default-branch tree and selects files whose normalized path:

- begins with the root `skills/` directory;
- ends with `/SKILL.md`;
- does not escape the root after path normalization;
- is not a template, fixture, benchmark, generated copy, hidden compatibility copy, or symbolic link.

Consequently, `skills/pdf/SKILL.md` is eligible, while `template/SKILL.md`, `.openclaw/skills/pdf/SKILL.md`, and `benchmarks/example-SKILL.md` are not. Nested Skill directories below `skills/` remain eligible when they have their own `SKILL.md`.

Each eligible manifest produces one `SourceRecord` and at most one Tool Card draft. Stable identity is derived from normalized repository full name plus normalized Skill directory path, not Skill display name, stars, rank, date, or default branch. Source URLs include the repository and the manifest path. Repository-level stars, license, topics, and activity are shared evidence; Skill name, description, instructions, declared dependencies, and boundaries come from that Skill's own manifest.

If two manifests declare the same display name, they remain distinct cards unless canonical repository and path also match. Existing duplicate and promotion checks still detect conflicts with already published IDs and URLs.

## Staged Crawl and Evidence

The crawler currently emits one snapshot per Source. P2 generalizes the internal crawl boundary so one enabled Source may emit an ordered set of snapshots:

1. one GitHub repository-search response;
2. one default-branch tree response for each of the two selected repositories;
3. one raw text response for every eligible `SKILL.md`.

The Source parser receives all successful snapshots for that Source and assembles Skill Source Records only when repository metadata, tree membership, and manifest content can be connected. Existing single-response parsers use the same grouped interface through a compatibility adapter and keep their output unchanged.

GitHub Search and tree endpoints use the public GitHub API. Manifest text is read from the public raw-content endpoint so enumerating all Skills does not consume one GitHub API content request per manifest. Raw responses are saved through the existing content-hash snapshot mechanism before parsing. The crawl audit records safe URL, status, hash, response metadata, selected repository, branch, and manifest path; it never records credentials.

Tree responses marked truncated are not treated as complete. The affected repository produces a visible partial/failure finding and no Skill candidates in that build. Individual missing, oversized, non-text, or malformed manifests are rejected independently without discarding valid sibling Skills.

Defensive limits apply to response bytes, normalized path length, per-manifest bytes, redirect count, and total Source bytes. Limits prevent resource exhaustion; they do not impose a normal business cap on the number of eligible Skills in the two selected repositories.

## Skill Parsing

The Skill parser treats repository text as untrusted input. It parses supported frontmatter and Markdown without evaluating templates, following embedded instructions, installing dependencies, or executing scripts.

The parser extracts or derives only evidence needed by the common Tool Card and Skill rating policy:

- display name and summary;
- repository and manifest URLs;
- primary purpose and use cases;
- explicit non-goals, limits, and failure guidance;
- supported agents or platform dependencies when stated;
- referenced scripts, references, and assets, checked against the fetched tree but not downloaded or executed;
- declared filesystem, network, shell, code-execution, browser, secret, or other permissions;
- repository maintenance, license, and trust signals;
- structural Skill signals such as presence of trigger guidance, actionable steps, boundaries, references, and platform-specific assumptions.

Absent evidence remains absent or unknown. Repository popularity must not fabricate trigger quality, instruction quality, permissions, boundaries, installation instructions, portability, or safety.

## Common Deterministic Review Findings

P2 introduces an internal finding shape used by common and type-specific validators:

```ts
interface ReviewFinding {
  code: string;
  severity: "info" | "warning" | "blocking";
  target: "missing_field" | "risk" | "evidence" | "human_review";
  message: string;
  evidence_refs: string[];
}
```

Common checks continue to cover evidence, required Tool Card fields, maintenance, safety clarity, feedback health, and duplicates. Skill checks add deterministic findings for manifest parsing, stable source identity, referenced-resource existence, unknown execution requirements, dangerous approval-bypass language, and unsupported or ambiguous dependencies.

The finding mapper writes into the existing Auto Review item fields:

- `evidence` -> `key_evidence` and `evidence_urls`;
- `missing_field` -> `missing_fields`;
- `risk` -> `key_risks`;
- `human_review` -> `human_review_reasons`.

`tool_card_auto_review.v1`, its `scorecard`, and its suggested-action vocabulary do not change. The refactor must freeze current non-Skill output through regression fixtures. Skill findings may change whether a new Skill draft is promoted, kept as draft, sent for review, or rejected, but they do not introduce a separate Skill review artifact or serialized `skill.v1` policy label.

## Common Rating Engine with Skill Policy

The rating engine gains one internal `RatingPolicy` boundary selected by `ToolCard.type`. Shared logic continues to own risk derivation, penalties, boosts, feedback adjustment, safety ceilings, recommendation level, explanations, and the common `rating_result.v2` envelope.

The Skill policy implements the authoritative weights in `docs/06-rating-rules.md`:

| Dimension | Weight |
| --- | ---: |
| `trigger_clarity` | 18 |
| `instruction_quality` | 20 |
| `task_fit` | 20 |
| `boundary_clarity` | 12 |
| `portability` | 10 |
| `evidence_quality` | 10 |
| `maintenance_health` | 5 |
| `security_posture` | 5 |

Skill dimension signals come from the normalized Tool Card plus deterministic Source Record evidence associated through `evidence_refs`. The Rating Result remains self-explanatory by citing those refs. Missing signals lower the corresponding dimension and confidence; stars and repository rank affect only bounded community/maintenance evidence and cannot substitute for Skill content.

The rating rules version advances to `rating_rules.v0.2` because Skill scoring semantics change. It is the build-wide ruleset version, so every newly generated Rating Result reports `rating_rules.v0.2`. Existing Skill cards are rerated with the Skill policy. All non-Skill cards use a compatibility policy that must reproduce current `rating_rules.v0.1-draft` dimension values, base scores, risk levels, recommendation levels, penalties, and boosts; the version advance for those records identifies the containing ruleset, not a scoring-semantic change.

Full MCP, Agent, Framework, CLI, and Prompt/Rules policies are later increments requiring their own before/after evidence. P2 does not silently claim those documented type weights are newly implemented.

## Failure and Authoritative-Snapshot Behavior

This Source does not use `previousSourceRecords` as fallback. Its output is the set of valid Skill records assembled from successful responses in the current build:

- A repository-search failure produces zero records for this Source and emits crawl failure evidence.
- A tree failure or truncation omits every Skill from that repository while allowing another successfully enumerated top-two repository to continue.
- An individual missing, oversized, non-text, malformed, or failed manifest omits only that Skill while valid sibling manifests continue.
- A change in top-two ranking removes cards from repositories that leave the current top two and adds candidates from newly selected repositories.
- A successfully fetched tree that no longer contains a previously published manifest removes that Skill from this Source's current output.
- Invalid manifests, path traversal, symbolic links, unsupported binary content, size-limit violations, or unresolved dangerous behavior cannot auto-promote and do not reuse an older record.
- The Source may produce zero or many eligible Skills. It never fabricates records to reach a target count.

Every omitted repository or manifest remains visible in crawl audit, parser warnings, review evidence, and current/previous data-quality comparison where applicable. Visibility does not convert failure into fallback. The user explicitly approved this potentially subtractive behavior for the bounded `github-topic-agent-skills` Source; other Source failure policies remain unchanged.

## Security Boundaries

- Only public GitHub Search, repository tree, and raw text endpoints are accessed.
- No Authorization header, cookie, private repository, browser session, local filesystem source, or user credential is used.
- Repository text, frontmatter, links, and instructions are data. They never become agent directives.
- The crawler does not clone repositories, initialize submodules, follow arbitrary external links, download referenced executables, install packages, or run scripts.
- Redirects are restricted to expected HTTPS GitHub-owned hosts and bounded in count.
- Logs and artifacts contain only safe request metadata already allowed by the crawl audit.
- Topic popularity is a discovery signal, not a trust decision. Every Skill passes validation, Auto Review, release admission, and promotion checks.

## Approval Record

The user approved direct development on `main`, a shared Tool Card and Rating Result storage model, a common core plus type-specific rating policies, unchanged Auto Review artifact structure, Skill-specific deterministic findings mapped to existing fields, one Tool Card per `SKILL.md`, and use of all eligible Skills from the top two `topic:agent-skills` repositories. The user further selected current-build authoritative-snapshot semantics: keep only successfully fetched current records, do not merge or fall back to the previous build, and allow ranking or crawl changes to remove old cards from this Source. Large Skill rating-rule changes and this potentially subtractive Source behavior are therefore approved for this bounded P2 scope; other type policies are not.

During final local verification, repeated anonymous builds exhausted the workstation's GitHub API quota. The user approved a one-run, process-local `gh` transport shim. It added authorization only to public `api.github.com` requests, was deleted immediately after the build, and was never committed or written to artifacts. The selected repositories were explicitly verified as public. This operational exception did not add authentication support to the product crawler or CI and does not change the approved public, unauthenticated production boundary.
