## Background

Agent Radar already exposes reviewed tool knowledge through Web, static artifacts, HTTP, and a remote MCP server. Coding agents can use those surfaces, but the repository does not yet ship a portable Agent Skill that teaches an agent when and how to query Agent Radar, how to interpret its action boundary, or how to keep recommendation credentials out of ordinary input.

The open Agent Skills repository convention discovers `skills/**/SKILL.md`, while Codex can also install a Skill from a GitHub directory URL. A repository-owned Skill therefore adds a low-infrastructure distribution surface without creating an installation marketplace or changing the production API.

## Goals

1. Ship `skills/agent-radar` as a valid, installable Agent Skill with concise trigger metadata and Codex UI metadata.
2. Provide a dependency-free Node client for version, search, Tool Card lookup, rating explanation, and task-oriented recommendation through the existing production HTTP API.
3. Preserve the existing read-only decision boundary: the Skill never installs, authorizes, or executes a recommended tool.
4. Keep provider credentials request-scoped and out of JSON, command arguments, files, logs, and output.
5. Document generic and Codex-specific installation paths and protect the Skill contract with automated tests and the standard Skill validator.

## Non-Goals

- Installing or executing third-party tools recommended by Agent Radar.
- Publishing a paid marketplace, package registry, new hosted service, or background daemon.
- Replacing the remote MCP distribution or changing HTTP/MCP schemas.
- Bundling a provider credential, persisting one, or accepting `api_key` in Skill command JSON.
- Claiming compatibility with agents that do not implement the basic `SKILL.md` convention.

## Design

The repository owns one public Skill at `skills/agent-radar`. `SKILL.md` uses the common minimal frontmatter contract and teaches a search-first workflow: discover candidates, inspect Tool Cards and rating evidence, use dynamic recommendation only when task context warrants it, then preserve the deterministic `use`, `compare`, `ask_human`, `avoid`, or `no_reliable_match` boundary.

`scripts/agent-radar.mjs` uses Node's built-in Fetch API and no package dependency. It defaults to the production Worker, permits an HTTPS base URL override and loopback HTTP for tests, accepts structured JSON for search and recommendation, and sends an optional provider key only from `AGENT_RADAR_LLM_API_KEY` to `X-Agent-Radar-LLM-API-Key`. It rejects legacy body credentials before network access and does not echo request input on failures.

The public README documents installation with the open `skills` CLI and a direct GitHub directory URL for Codex. Installation does not imply authorization to call sensitive systems: Agent Radar itself remains a read-only decision aid, and project policy plus human confirmation still govern any later tool adoption.

## Acceptance

1. The standard Skill validator accepts `skills/agent-radar`.
2. The open Skills CLI can discover the repository Skill by name from a local checkout.
3. Contract tests prove search routing, response parsing, and header-only recommendation credentials.
4. The client rejects `api_key` in JSON without echoing its value.
5. `npm test`, `npm run lint`, `npm run stylelint`, `npm run pages:build`, and `git diff --check` pass.
6. README, product requirements, workflow, architecture, evaluation, security, operations, and Roadmap authorities describe the new surface without changing the API or execution boundary.

## Risks

- The Skill depends on production API availability; failures must remain explicit and must not produce invented recommendations.
- Generic Skill installers and agent-specific discovery paths may evolve. The Skill keeps only common frontmatter as its portability boundary and documents both a generic and Codex path.
- Users may mistake recommendation for permission to install. Repeated Skill instructions and security documentation explicitly prohibit that inference.
- Passing sensitive project context in a natural-language task could expose it to the Worker or provider. The Skill requires minimum non-sensitive task descriptions.
