# v0.9 P2: Local-First Versioned Skill Dataset Design

- Status: Completed
- Implementation commits: `0846e592`
- Current status source: [docs/15-roadmap.md](../v0.x-roadmap.md)

## Background

The initial v0.9 P1 draft wrapped the production HTTP API in an installable Skill. That made Agent Radar easier to invoke but substantially overlapped the existing MCP surface and still required an online request for every query. P2 replaces that uncommitted adapter with a local-first dataset model: the Skill synchronizes a compatible reviewed catalog, verifies it, and performs search and context assembly locally. The host agent uses the Skill instructions and local evidence for recommendation reasoning.

## Goals

1. Publish a stable v1 Skill dataset channel and one immutable release directory per reviewed Agent Radar release.
2. Include Tool Cards, Rating Results, and Search Index with exact schema, byte-size, and SHA-256 declarations.
3. Preserve prior v1 release directories in later reviewed bundles after revalidating their manifests and contents.
4. Make the Skill client download into a temporary same-filesystem directory, validate compatibility and every file, write a verification marker, then atomically switch `current.json`.
5. Keep the last verified release active when download, compatibility, schema, size, or checksum validation fails.
6. Perform search, Tool Card lookup, rating explanation, and recommendation-context construction locally without MCP or a provider credential.

## Non-Goals

- Running the full hosted LLM recommendation engine inside the Skill.
- Installing, authorizing, or executing recommended third-party tools.
- Background updates, install hooks, hidden network access, or a credential store.
- Signing the dataset manifest in P2; HTTPS, fixed-origin checks, reviewed-bundle checksums, and release provenance are the current authenticity boundary.
- Supporting dataset contract v2 before an incompatible change exists.

## Data Contracts

`agent_radar_skill_channel.v1` is the mutable v1 pointer at `/data/skill/channels/v1/latest.json`. It declares `agent_radar_skill_dataset.v1`, the release ID, and the exact path, size, and SHA-256 of that release's manifest.

`agent_radar_skill_data_manifest.v1` is stored at `/data/skill/releases/<release-id>/manifest.json`. It declares the dataset contract, minimum client version, release and commit identity, data version, publication time, and exactly three files: `tool_cards.jsonl`, `ratings.jsonl`, and `search_index.json`.

Ordinary data changes create a new release manifest without changing contract versions. An incompatible field, file-layout, or semantic change requires a new dataset contract major and a separate channel. Old clients remain on the last verified compatible release.

## Publication and Retention

The pipeline generates the current release from the same Tool Cards, Ratings, and Search Index used by Web, HTTP, MCP, and evaluation. When a previous reviewed bundle is available, it validates every previous Skill release manifest, size, checksum, and directory identity before copying it into the new bundle. The current release ID cannot overwrite an inherited release directory.

This preserves immutable v1 release URLs across deployments. Bundle growth is initially accepted because each release adds three compact data files; file count and static-asset size must be monitored before introducing a reviewed retention policy or object storage.

## Client Synchronization

The client supports manifest/channel v1 and dataset v1 with client version `0.9.0`. `sync`:

1. Fetches the fixed v1 channel from the configured HTTPS origin, allowing HTTP only for loopback tests.
2. Validates schema, contract, release ID, manifest path, origin, size, and checksum.
3. Fetches and verifies the release manifest and its minimum client version.
4. Restricts files to the exact allowlist, per-file 25 MiB, and total 50 MiB.
5. Downloads into `releases/.<release>.tmp-<uuid>`, verifies size, SHA-256, JSON/JSONL parsing, and record schema.
6. Writes `_verified.json`, atomically renames the release directory, then atomically renames a temporary `current.json` pointer.

Local queries read `current.json` once and use only that release directory. Failed synchronization removes the temporary directory and does not modify the current pointer.

## Local Recommendation Context

`context` performs local search and returns matching Tool Cards, Ratings, task input, release provenance, and a deterministic `maximum_allowed_action` for each candidate. Unknown-trust code or shell execution caps at `avoid`; high, critical, unknown, explicit human-approval, or tolerance-exceeding risk caps at `ask_human`. The host model may be more conservative but must not exceed that ceiling. Empty results mean `no_reliable_match`.

## Acceptance

1. Pipeline and unit tests prove channel/manifest identity, checksums, prior-release preservation, and tamper rejection.
2. Client tests prove atomic synchronization, offline search, local context safety ceilings, and old-release fallback after checksum failure.
3. The standard Skill validator and local installer discovery pass.
4. Public documents describe the local-first boundary and compatibility policy.
5. `npm test`, `npm run lint`, `npm run stylelint`, `npm run pages:build`, and `git diff --check` pass.

## Risks

- SHA-256 detects corruption and mismatch but does not independently authenticate a maliciously replaced channel plus dataset. A signed manifest remains a future hardening option.
- Retaining all v1 releases grows the reviewed bundle. Monitor asset count and size before setting a retention policy.
- Local prompt-based recommendation may vary by host model. Deterministic per-candidate action ceilings reduce but do not eliminate output variance.
- Atomic rename requires temporary and final paths on the same filesystem; the client always creates both under one cache root.
